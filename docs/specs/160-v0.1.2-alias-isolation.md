# 160 — v0.1.2 Alias Isolation

> Status: **proposed**. Second release in the three-release sequence
> that lands the contracts in [`140-v0.2-operability.md`](140-v0.2-operability.md).
> Companion plan: [`../plans/020-v0.1.2-alias-isolation.md`](../plans/020-v0.1.2-alias-isolation.md).

## 1. Summary

v0.1.2 is a non-breaking patch that does exactly one thing: remove
`rusqlite-bundled` from the workspace and replace `aliases.sqlite`
with an `fd-lock`'d `aliases.json` file. No other changes.

The release has two purposes, in priority order:

1. **Forensic**: isolate the rusqlite removal as a single commit
   so we can cleanly bisect whether rusqlite-bundled is the cause
   of the macOS release-mode host-client SIGABRT documented in the
   v0.1.0 CHANGELOG. If the crash disappears on v0.1.2, hypothesis
   confirmed. If it persists, hypothesis falsified — and v0.2.0's
   larger diff will not contaminate the bisect.
2. **De-risking v0.2**: §9 (the alias-store change) is one of the
   items with measurable binary-size and build-time impact and one
   of the few with a real behaviour change (file-level lock vs
   SQLite WAL). Shipping it alone validates the new semantics
   before they're bundled with everything else.

## 2. The hypothesis being tested

### 2.1 The crash, as observed

From `CHANGELOG.md` v0.1.0 entry:

> **macOS release-mode host-client SIGABRT.** `portl exec` against
> a real ticket, running a **release** build on macOS arm64,
> aborts with `libmalloc: double free` inside
> `peer::connect_peer`. Debug builds are unaffected. Linux
> (both libc and musl) is unaffected. Exact reproduction: see
> `scratch/2026-04-17-mac-release-crash.md`.

### 2.2 Working hypothesis

`rusqlite-bundled` is implicated because:

- The crash is release-mode only — consistent with a
  compiler-optimization interaction with sqlite's amalgamation
  C thread-local-storage setup.
- The crash is inside `libmalloc` — consistent with sqlite's
  custom allocator or its interaction with rust's allocator.
- The crash is macOS-only — consistent with Darwin's stricter
  `libmalloc` double-free detection; other platforms would silently
  corrupt or succeed.
- `rusqlite-bundled` is the largest transitive native-C dependency
  in the workspace.

### 2.3 Hypothesis we are not willing to assume

`connect_peer` itself does not call into the alias store when
invoked with a raw ticket
(`crates/portl-cli/src/commands/peer.rs:31-72`). The alias store is
touched only when the user supplies an alias string. Which means:

- If the crash **reproduces** with a raw ticket, then alias-store
  code cannot be on the stack at crash time, and the
  rusqlite-linked binary artifact is the only remaining smoking
  gun. v0.1.2's dep-graph cleanup is the right experiment.
- If the crash **only** reproduces with an alias-resolved ticket,
  then `rusqlite-bundled` is directly implicated on the hot path
  and v0.1.2 is equally well-targeted.

Either way, v0.1.2 is the right experiment. **But we also need a
fallback**: if v0.1.2 ships and the crash persists with a raw
ticket, the real cause is elsewhere (iroh, rustls, or crypto).
The §5 test protocol below explicitly reproduces both cases before
drawing conclusions.

## 3. Contracts this release establishes

### 3.1 No CLI, config, or env-var changes

v0.1.2 is byte-wise compatible at the user interface with v0.1.1.
The only observable differences are:

- Binary is smaller (expected ≈ 800 kB–1.2 MB).
- Cold-build time is faster (expected ≈ 20 % reduction).
- The `$PORTL_HOME/aliases.sqlite` file is migrated on first
  v0.1.2 startup (see §3.3).
- No release-mode SIGABRT on macOS arm64 *if the hypothesis is
  correct*.

### 3.2 Alias store: `aliases.json` with `fd-lock`

File: `$PORTL_HOME/aliases.json` (UTF-8, JSON, pretty-printed for
grep-friendliness).

Schema:

```json
{
  "version": 1,
  "aliases": {
    "<alias>": {
      "endpoint_id": "<32-byte hex>",
      "ticket": "<base32-lowercase URL-safe canonical>",
      "created_at": "<RFC3339>",
      "updated_at": "<RFC3339>"
    }
  }
}
```

Concurrency: `$PORTL_HOME/aliases.json.lock` is the companion
file used by `fd-lock`. Readers take a shared lock; writers take
an exclusive lock for the full read-modify-write cycle. On
`ENOSPC` during the `fs::rename(tmp, final)` step, the old file is
left intact and the write returns an error the caller surfaces.

**Semantics vs v0.1 SQLite WAL:**

| Property | v0.1 (SQLite WAL) | v0.1.2 (JSON + fd-lock) |
| --- | --- | --- |
| Corruption under concurrent writers | no | no |
| Lost-write under concurrent read-modify-write | possible (separate connections) | possible (same race, documented) |
| Lock granularity | per-row / per-statement | whole file |
| Read during write | yes (WAL) | yes (shared lock waits for writer) |
| Durability of committed writes | fsync on commit | fsync on `rename` |

The coarser locking is a deliberate choice: the alias store has ≤
1 kHz write rate (human-entered commands), not a workload where
WAL's per-row granularity matters. Documented in CHANGELOG.

### 3.3 One-shot migration on first v0.1.2 startup

If `$PORTL_HOME/aliases.sqlite` exists and `$PORTL_HOME/aliases.json`
does not:

1. Open `aliases.sqlite` read-only via the last embedded
   rusqlite-link (kept in v0.1.2 *only* for this migration path,
   then dropped in v0.2).
2. Read every row; convert to the v1 JSON schema above.
3. Write `aliases.json.tmp` with exclusive `fd-lock`; `fsync`;
   `rename` to `aliases.json`.
4. Rename `aliases.sqlite` to `aliases.sqlite.migrated-<timestamp>`.
   The operator can delete it at their leisure.
5. Log the migration summary (alias count, byte sizes) at `info!`.

On subsequent startups, only the JSON file is read; the migration
code does nothing.

**Exception:** if both files exist, v0.1.2 errors out with a loud
diagnostic instead of silently picking one. Operator resolves
manually.

### 3.4 Why `rusqlite` stays linked in v0.1.2 (contradicts the
### release's own premise, briefly)

v0.1.2 keeps `rusqlite` in the dep graph *purely* for the
migration-read path above. This means the binary is **only
partially smaller** than v0.2's target — most of `rusqlite-bundled`
and its build-script dependencies are still present.

This means the v0.1.2 bisect experiment has a subtle confound:
the release mode code still links rusqlite (just doesn't call into
it on the hot path). If the macOS crash **disappears** in v0.1.2,
that confirms rusqlite's compiled-in presence on the hot path is
the issue (either through alias-store calls or through
link-order-dependent `libmalloc` behaviour). If the crash
**persists**, the link-only hypothesis is falsified too, and v0.2
will drop rusqlite entirely (migration path is removed after
v0.1.2) to complete the elimination.

The migration path is acceptable for one release: it reads a file,
parses it, writes JSON, and never runs again. No concurrent-writer
interactions; no long-lived sqlite connections.

## 4. Acceptance gates

A v0.1.2 build is releasable when all of the following hold.

### 4.1 Functional

1. Fresh host + v0.1.2: `portl id new && portl exec alias-name`
   (with an alias created via `portl ... --save-alias`) works
   end-to-end, using the JSON alias store.
2. Host with v0.1.1's `aliases.sqlite` + no `aliases.json`:
   v0.1.2 first startup migrates; `aliases.sqlite.migrated-<ts>`
   exists; `aliases.json` exists and contains the expected rows;
   second startup does not re-migrate.
3. Host with both files present: v0.1.2 refuses to start, prints a
   clear resolution message, exits non-zero.
4. Concurrent writers test: two processes both call the public
   alias-store write API 1000 times; no file corruption, no
   deadlock, both exit 0, final file is valid JSON.

### 4.2 Dependency graph

5. `cargo tree --workspace -e normal` shows `rusqlite` present but
   **only** as a dependency of `portl-cli` (not `portl-agent`).
6. `cargo tree --workspace -e normal` shows `libsqlite3-sys`, `cc`,
   `cmake` still present (they come back in with rusqlite).
7. v0.2.0 **does not** regress to keeping them; this is a one-
   release-only carryover.

### 4.3 Forensic

8. macOS arm64 release build: `portl exec <raw-ticket>` against a
   vn3 target 50 times in sequence; zero SIGABRTs. If this fails,
   the rusqlite hypothesis is falsified; see §5.
9. macOS arm64 release build: `portl exec <alias-resolved-ticket>`
   against a vn3 target 50 times in sequence; zero SIGABRTs.
10. Linux x86_64 musl release build: both gates 8 and 9 still
    pass (no regression).

### 4.4 No regressions

11. Full `cargo test --workspace --all-features` passes on both
    platforms; count = v0.1.1's count + new tests only.
12. `portl doctor` on a v0.1.1 `$PORTL_HOME` (post-migration)
    succeeds.

## 5. If the crash persists: falsification protocol

If gates 4.3.8 or 4.3.9 fail on v0.1.2, the rusqlite hypothesis is
falsified (at least in its current form). Do not ship v0.1.2 as-is;
instead:

1. Capture a crash dump with Rust's `RUST_BACKTRACE=full` and
   `MallocStackLogging=1`.
2. Re-run under `lldb` with `break set -n malloc_error_break` to
   catch the `libmalloc` abort site.
3. Compare the backtrace against v0.1.0's.
4. Open a new investigation doc in
   `scratch/2026-04-XX-mac-crash-post-rusqlite.md` with the
   evidence.
5. Pivot: the next suspects are
   - `iroh`'s QUIC transport init (release-mode rustls codegen
     oddities),
   - `ed25519-dalek`'s feature flag selection in release,
   - the `portable_pty` macOS-specific code path,
   - rustls's choice of aws-lc-rs vs ring under release.
6. v0.1.2 is **still shippable** without the crash fix, because
   the alias-store simplification and dep-graph cleanup are
   independent wins; the macOS crash just stays on the known-issue
   list and moves to v0.2.0's critical path.

## 6. Risks and mitigations

| Risk | Severity | Mitigation |
| --- | --- | --- |
| Migration path itself triggers the macOS crash (because rusqlite is still linked) | HIGH | The migration runs once, synchronously, before the endpoint is even created. If it fails, no session work has happened yet and the failure is diagnostic. |
| Operator has a `aliases.sqlite` locked by a concurrent v0.1.1 process during upgrade | MEDIUM | Upgrade docs require stopping v0.1.1 before starting v0.1.2. v0.1.2 checks and refuses if the sqlite file is locked. |
| `fd-lock` is unreliable on NFS / SMB / ZFS-on-other-platform | MEDIUM | `fd-lock` uses `fcntl(F_SETLK)` on POSIX. NFSv4 advisory locks are documented as "use at your own risk"; CHANGELOG notes `$PORTL_HOME` on network FS is unsupported. |
| Hypothesis is confirmed but only for alias-resolved tickets (§2.3 branch 2) | LOW | §4.3 gates 8 and 9 differentiate. Falsification protocol (§5) still applies to the gate that failed. |
| JSON file gets too large | LOW | Alias entries are small; at 10 MB the file still parses in < 10 ms. v0.2 adds no ceiling here (revocations.jsonl gets one; aliases do not). |

## 7. Release mechanics

Tag: `v0.1.2`. Target: commit on `main` containing the plan's
final "Commit" step.

Artifacts: same four tarballs as v0.1.1, same packaging
(`zstd -19`), same sha256 sidecars. Expect linux musl tarball to
be ≈ 800 kB smaller; macOS arm64 ≈ 1.2 MB smaller.

CHANGELOG: entry for v0.1.2 with explicit forensic framing —
"released to test the rusqlite-bundled hypothesis for the macOS
release crash. Outcome: confirmed / falsified. Next step: …"

Post-release verification is **the experiment**: §4.3 gates on
max (macOS arm64 M3) running a fresh `portl exec` against vn3 50
times. Results logged to
`scratch/2026-04-XX-v0.1.2-mac-bisect.md` regardless of outcome.
