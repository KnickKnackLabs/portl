# 170 — v0.3.3 Relay: Self-Host MVP with Peer-Store Authentication

> Status: **draft**, targeting v0.3.3. Depends on v0.3.2 observability
> release (for `portl status network` + `portl status relay`).
> Companion spec: [`180-v0.3.4-peer-pairing.md`](180-v0.3.4-peer-pairing.md)
> adds relay hints to peer entries and the pair-protocol relay-URL
> exchange.

## 1. Summary

v0.3.3 lets any portl-agent optionally serve as a relay for its
peers. The relay runs in-process alongside the agent, shares its
identity and peer store, and authenticates incoming relay clients
by endpoint_id against that peer store. Wire-compatible with iroh's
existing relay protocol; closed-by-default on the access side.

Two shipping outcomes:

1. `portl-agent --relay-enable` turns the agent into a relay for its
   peers — no extra process, no extra identity, no extra config
   plane.
2. `portl status relay` reports bind addr, TLS cert lifecycle,
   active connections, and rejection counts.

Explicit non-goals:

- Relay federation, pool selection, or multi-relay fallback policy —
  deferred to v0.4.
- Per-peer rate limits or bandwidth caps.
- Abuse reporting / quota enforcement.
- Running a hosted relay operated by KnickKnackLabs.

## 2. Motivation

Current state (v0.3.1):

- portl defaults to `RelayMode::Disabled`. Without a relay, any peer
  pair behind symmetric NAT is unreachable.
- `PORTL_DISCOVERY=relay:<url>` configures a single relay. No CLI,
  no multi-relay, no peer-side hints.
- iroh's n0-operated public relays exist but require `presets::N0`
  and implicitly tie portl's availability to n0 infrastructure.
- The iroh-relay binary is open source but has no portl-level
  authentication. Run one and anyone on the internet can use it.

Three unmet needs this release addresses:

1. **Self-host without ops burden.** Operators running a homelab
   already have a machine; they want that same machine to relay for
   peers it already trusts. No second binary, no second service
   definition, no second identity to manage.
2. **Privacy.** Traffic metadata (who's talking to whom, how much,
   when) shouldn't flow through a third-party relay by default.
3. **Closed-by-default.** Running a relay shouldn't mean serving
   strangers. Relay access should piggy-back on the existing peer
   trust model — if you're in the operator's peer store, you can
   use their relay; otherwise the connection is rejected at
   accept-time.

## 3. Design decisions

### 3.1. Relay lives in the agent process

One binary, one process, one identity. The relay listener is added
to the existing agent startup alongside the portl-protocol UDP
listener. Shares `AgentState` — same `TrustRoots`, same peer-store
reload task, same logging subscriber.

Rejected alternative: separate `portl-relay` binary. Reasons:

- Duplicates identity/trust plumbing.
- Doubles the service-management surface (plist/unit per process).
- Splits reload behavior (adding a peer would need to reach both).
- Most operators who run a relay already run the agent. Separate
  binary is strictly more overhead.

### 3.2. Peer store IS the access control list

Authorization at relay connect-time:

```text
client connects → TLS handshake → iroh-relay WS accept →
    lookup client.endpoint_id in PeerStore →
    if present AND not held → allow
    else → close with policy-rejected frame
```

No new schema, no new cap type, no ticket validation on the relay
path. `portl peer add-unsafe-raw <eid> --mutual` grants relay
access exactly the same way it grants ticket-handshake access.

Rejected alternatives:

- **Relay-specific tickets** (`relay:*` cap, time-bounded). Useful
  but requires a new cap grammar + ticket validation at the relay
  accept path + issuance UX. Deferred to v0.4; peer-store membership
  covers the common case.
- **Separate relay allowlist file** (e.g. `relay_allow.json`).
  Forces operators to maintain two ACLs (peer store + relay
  allowlist). Merge into one.

### 3.3. Three policy tiers, opt-in

| Policy | Behavior | Default |
|---|---|---|
| `open` | Accept any iroh endpoint (matches standard iroh-relay) | no |
| `peers-only` | Accept only endpoints in `peer_store` | **yes** |
| `pairs-only` | Accept only peers with `accepts_from_them=true` AND only forward between pairs both authorized by operator | no |

`peers-only` is the default when `--relay-enable` is set. Matches
portl's "closed by default" posture elsewhere (empty peer_store
rejects all tickets; fail-closed on missing identity; etc.).

`pairs-only` is stricter but not enforced in v0.3.3 — it flags in
the status output as "configured but enforcement requires v0.3.4
peer pair protocol" and falls back to `peers-only` behavior until
v0.3.4 lands.

### 3.4. Wire compatibility

Relay clients speak iroh-relay's existing WebSocket-over-TLS
protocol. Zero changes to iroh on the client side. Portl-relay is
a closed-policy variant of iroh-relay, not a fork.

Peers using a portl-relay configure it identically to any iroh
relay:

```bash
PORTL_DISCOVERY=relay:https://relay.mynet.com
```

Relay URL distribution in v0.3.3 is manual (operator shares URL
out-of-band). v0.3.4's pair protocol automates this via peer
entries carrying a `relay_hint` field.

### 3.5. TLS certificate handling

Relay requires a TLS cert. Three supported modes:

1. **Operator-provided cert** (production): pass `--relay-cert` +
   `--relay-key` paths. Operator manages LE/ACME renewal
   externally. Portl watches files for mtime change; reloads on
   next mtime touch.
2. **ACME on startup** (convenience): pass `--relay-acme-email
   <email> --relay-hostname <dns-name>`. Agent uses
   `rustls-acme` crate to obtain + renew a LE cert. Cert cached
   at `$PORTL_HOME/relay-cert/`.
3. **Self-signed for testing** (dev): pass `--relay-selfsigned`.
   Generates a one-shot self-signed cert pinned to the agent's
   hostname. Clients must add `--relay-insecure-selfsigned` when
   dialing. Makes it obvious this is non-production.

v0.3.3 ships modes 1 and 3. Mode 2 (ACME) deferred to v0.3.3.1 or
v0.3.4 depending on `rustls-acme` review.

## 4. CLI surface

### 4.1. Enabling the relay

```bash
# Explicit install with relay
portl install --apply --yes \
  --with-relay \
  --relay-bind 0.0.0.0:443 \
  --relay-policy peers-only \
  --relay-cert /etc/portl/relay.crt \
  --relay-key  /etc/portl/relay.key \
  --relay-hostname relay.mynet.com

# Ad-hoc run (no service install)
portl-agent \
  --relay-enable \
  --relay-bind 0.0.0.0:443 \
  --relay-cert /etc/portl/relay.crt \
  --relay-key  /etc/portl/relay.key

# Toggle later by re-running install --apply with/without --with-relay.
# install rewrites the plist/unit; re-kicks the service.
```

## 5. Implementation plan

### 5.1. Module layout

```
crates/portl-agent/src/
  relay/
    mod.rs           # re-exports, configuration glue
    listener.rs      # iroh-relay wrapping + connect-gate
    policy.rs        # Open | PeersOnly | PairsOnly
    cert.rs          # cert loading + mtime reload
  lib.rs             # gains spawn_relay_task when enabled
  config.rs          # gains RelayConfig struct + env parsing

crates/portl-cli/src/commands/
  install/
    apply.rs         # --with-relay + relay-* flags emit plist/unit
    render.rs        # plist/unit templates gain ProgramArguments
    mod.rs           # accept + thread through
  status/
    relay.rs         # NEW — reads metrics.sock, renders relay section
```

### 5.2. Agent-side wire integration

```rust
// crates/portl-agent/src/relay/listener.rs
pub(crate) async fn serve(
    cfg: RelayConfig,
    state: Arc<AgentState>,
    shutdown: CancellationToken,
) -> Result<()> {
    let tls = load_tls_config(&cfg.cert_path, &cfg.key_path)?;
    let accept = |req: &ConnectRequest| -> bool {
        let client_eid = req.peer_endpoint_id();
        match cfg.policy {
            Policy::Open => true,
            Policy::PeersOnly => {
                let roots = state.trust_roots.read().unwrap();
                roots.0.contains(&client_eid)
            }
            Policy::PairsOnly => {
                // v0.3.3 treats same as PeersOnly; v0.3.4 adds
                // pair-table enforcement.
                let roots = state.trust_roots.read().unwrap();
                roots.0.contains(&client_eid)
            }
        }
    };
    iroh_relay::serve(bind, tls, accept, shutdown).await
}
```

The `accept` hook is the only portl-side gate; the remainder of the
relay wire is iroh-relay as-is.

### 5.4. Install-time changes

`portl install --apply --with-relay` rewrites the plist/unit to:

- Add `--relay-enable` and relay flags to `ProgramArguments` /
  `ExecStart`.
- Grant `NET_BIND_SERVICE` capability on Linux (systemd
  `AmbientCapabilities=CAP_NET_BIND_SERVICE`) so agent can bind :443
  without root.
- On macOS launchd, add `SessionCreate` + relay-appropriate keep-alive.
- Pin `UserName` / `Group` to the same non-root user; no privilege
  escalation for relay vs. no-relay.
- Validate the cert + key paths exist and are readable by the
  service user before writing the unit.

## 6. Testing plan

### 6.1. Unit tests

- `RelayPolicy::check_allowed(eid, &trust_roots)` behavior for each
  tier + edge cases (held peer, self, unknown).
- Cert loading: valid, invalid (expired, wrong key, unreadable).
- Reload on cert file mtime change.
- Env var parsing for all `PORTL_RELAY_*` vars + precedence vs. CLI
  flags.

### 6.2. Integration tests

- Two-endpoint loopback: E1 enables relay, E2 connects through it.
  Verify rejection when E2 is not in E1's peer store; verify
  acceptance when added.
- Live reload: add peer via `portl peer add-unsafe-raw`; verify next
  relay connect attempt succeeds within 500ms (reload-task
  window).
- Certificate mtime reload: rotate cert file; verify new connections
  use new cert; existing connections unaffected.

### 6.3. Smoke tests (CI)

- `portl install --apply --with-relay --relay-selfsigned` on a CI
  container; verify service starts; verify `portl status relay`
  reports enabled.
- Parse check for plist/unit with relay fields on both OSes.

## 7. Migration / back-compat

No breaking changes. Existing `PORTL_DISCOVERY=relay:<url>` continues
to work (client-side relay selection). New `PORTL_RELAY_*` vars are
server-side and orthogonal.

Agents upgrading from v0.3.2 → v0.3.3 with no relay flags behave
identically.

Peer store schema unchanged. (v0.3.4 adds `relay_hint` field in a
backward-compatible way.)

## 8. Security considerations

### 8.1. Trust boundary

The relay does NOT decrypt traffic — iroh's end-to-end encryption
means the relay operator sees connection metadata (endpoint_ids,
timing, bytes) but not payloads. This matches iroh-relay's threat
model.

Operator threat model: if you trust a peer enough to add them to
your peer store, you already trust them to initiate a shell / exec /
tcp forward. Letting them relay through your machine is a weaker
privilege (they can only forward traffic, not run code on your box).
So "peer ⇒ relay-allowed" is strictly safer than "peer ⇒ can-run-
commands."

### 8.2. Bandwidth amplification

A malicious peer could attempt to use the operator's relay for
arbitrary-destination traffic. Mitigations in v0.3.3:

- `peers-only` policy gates accept; only peers can join.
- iroh-relay's existing per-connection limits apply.
- `portl_relay_bytes_forwarded_total` + `portl status relay` give
  operators visibility.

v0.4 will add: per-peer bandwidth caps, rate limits on connection
establishment, optional bandwidth quotas.

### 8.3. Cert key handling

TLS private key must be readable by the service user (`thinh` or
`portl`) and no one else. Install validates mode-0400 / mode-0600
before writing the unit. Plist / systemd unit does NOT embed the
key content — only file paths.

### 8.4. Port :443

Binding :443 requires elevated capability. Three options:

1. **Root launchd agent** (macOS system daemon): runs as root.
2. **User launchd agent + port forward**: listen on :4443, pf-rule
   forward :443 → :4443. Operator sets up pf.
3. **systemd with CAP_NET_BIND_SERVICE**: non-root service binds
   :443 directly via ambient capability.

Install detects the OS + user-vs-root context and emits the right
template. Default suggestion for first-time operators: :4443 + doc'd
pf/iptables rule, avoids the cap-related footgun.

## 9. Open questions

1. **Self-signed cert in CI smoke**: do we ship a test-only
   self-signed cert in the repo, or generate on-the-fly in the
   smoke job? Leaning: generate on-the-fly (no checked-in secrets,
   even dummy ones).
2. **Relay liveness probe**: should the agent expose an HTTP
   `/health` on the relay port for LB health checks? v0.3.3 says
   no (iroh-relay's own accept path serves as a liveness signal);
   revisit if operators report need.
3. **Local discovery of home-relay**: when the agent enables a
   relay, should it advertise itself as its own home-relay so
   clients-of-this-agent prefer this relay? Probably yes; defer to
   v0.3.4 integration with peer pair protocol.
