# 165 — v0.3.2 Observability: Unified Status + JSON + Watch

> Status: **draft**, targeting v0.3.2. Ships between v0.3.1 (install
> / doctor hardening) and v0.3.3 ([relay MVP](170-v0.3.3-relay.md)).
> Companion spec: [`180-v0.3.4-peer-pairing.md`](180-v0.3.4-peer-pairing.md)
> consumes `portl status peers --active` for post-pair verification
> flows.

## 1. Summary

v0.3.2 consolidates observability into a single verb — `portl
status` — with focused subsections, `--json` output across the
read-only surface, and `--watch` for ambient dashboards. Splits
the responsibilities of `doctor` (healthcheck, pass/fail) from
`status` (informational dashboard, always 0).

Three shipping outcomes:

1. `portl status` as a unified dashboard with focused subsections
   (`identity`, `agent`, `service`, `peers`, `tickets`,
   `connections`, `network`, `package`).
2. `--json` flag across read-only verbs (`doctor`, `peer ls`,
   `ticket ls`, `whoami`, `status *`) with a stable schema v1.
3. `--watch [N]` flag on dashboard + focused views for ambient
   live displays, zero additional dependencies.

Explicit non-goals:

- TUI (ratatui / ncurses). `watch portl status` is the zero-code
  path; `--watch` is the polish.
- Event-streaming `--follow` (e.g. tail connection events).
  Deferred to v0.4 — requires a new IPC channel and agent push.
- Remote peer/ticket-store introspection via `portl status
  <target>`. Deferred to v0.3.4 (requires `meta:info` RPC + pair
  protocol).
- Historical metrics storage. Prometheus scrape endpoint already
  exists; users aggregate externally.

## 2. Motivation

Current state (v0.3.1):

- `portl doctor` conflates healthcheck and dashboard — it prints
  passing checks too, which muddies its "alarm bell" role.
- `portl peer ls`, `portl ticket ls`, `portl whoami` each have their
  own ad-hoc TTY output; no consistent JSON surface.
- No way to see "what connections are active right now" without
  scraping the metrics socket and parsing OpenMetrics.
- `portl handshake <target>` exists but is one-shot and only
  reports success/fail — no connection-path detail.
- Live monitoring requires `watch -n 1 'portl doctor && portl peer
  ls'`, which fires a fresh process every tick. ~40ms per spawn;
  fine for debug but not a great default.

Four unmet needs this release addresses:

1. **Answer "what is portl doing right now?" in one command.**
   `status` with no args = dashboard; everything relevant in one
   screen.
2. **Consistent machine-readable output.** `--json` schema v1 across
   the read surface; stable for tooling and tests.
3. **Ambient displays.** `--watch` for tiles (i3bar, tmux status,
   personal dashboard).
4. **Verb hygiene.** Separate `doctor` (alarm) from `status`
   (dashboard) — each does one job well.

## 3. Design decisions

### 3.1. Verb division

| Verb | Role | Exit code | Output |
|---|---|---|---|
| `doctor` | Healthcheck — "what's wrong?" | 0 if healthy, 1 if issues | Only speaks when problems exist (`--verbose` to see passing checks) |
| `status` | Dashboard — "what's happening?" | Always 0 (unless IPC fails hard) | Always informational |
| `status <target>` | Reachability probe — "can I reach X?" | 0 if reached, 1 if not | Connection details |
| `peer`, `ticket`, `whoami` | Noun ops — manipulate resources | Standard | Focused on the noun |

Rejected alternative: collapse `doctor` into `status --check` flag.
Reason: `doctor` as a dedicated alarm-bell verb is useful in
scripts (`portl doctor || send-alert`). Flagging into `status`
blurs that.

### 3.2. Status sections

```text
portl status              → dashboard (all sections)
portl status identity     → just identity
portl status agent        → just agent IPC readout
portl status service      → launchd / systemd state
portl status peers        → alias for `portl peer ls --active`
portl status tickets      → alias for `portl ticket ls`
portl status connections  → iroh per-connection table
portl status network      → relay / NAT / discovery diagnostic
portl status package      → package manager + binary drift
portl status <target>     → reachability + local view + remote view
```

Focused sections are substrings of the dashboard. Running `status
peers` ≡ running `status` and reading only the peers section, just
cheaper and less noise.

### 3.4. `--json` schema v1

Top-level envelope:

```json
{
  "schema": 1,
  "kind": "status",
  "generated_at": "2025-11-14T23:42:01Z",
  "identity": { ... },
  "agent":    { ... },
  "service":  { ... },
  "peers":    { ... },
  "tickets":  { ... },
  "connections": { ... },
  "network":  { ... },
  "package":  { ... }
}
```

Focused sections omit the keys they don't touch; `status peers
--json` returns `{"schema": 1, "kind": "status.peers", ...}` with
only the `peers` field populated. Consumers can union-decode.

`schema: 1` is the wire-compat promise. Additive fields are a minor
bump (still schema: 1); breaking changes bump to 2.

Error shape in `--json` mode:

```json
{
  "schema": 1,
  "kind": "error",
  "error": {
    "code": "agent_unreachable",
    "message": "metrics.sock connect failed: Connection refused",
    "hint": "is the agent running? try `portl status service`"
  }
}
```

Written to stderr, exit 1. Progress/status lines (e.g. "connecting
to peer...") are elided in `--json` mode.

### 3.5. `--watch [N]`

```rust
// ~30 LOC total
loop {
    print!("\x1b[2J\x1b[H");  // clear + home
    render_status()?;
    if !args.watch_continue_on_error && last_error { break; }
    tokio::time::sleep(Duration::from_secs(n)).await;
}
```

- `--watch` defaults to N=2 seconds.
- `--watch N` with explicit N (u64 seconds; minimum 1, maximum
  3600).
- `--watch --json` is rejected — watching JSON is meaningless.
- Ctrl+C exits cleanly with exit 0.
- On IPC error, prints the error in-line and keeps watching
  (transient agent restarts shouldn't kill the dashboard).

### 3.6. `portl status <target>` scope

Shape:

```text
$ portl status onyx
target:         onyx (d65f9e65…7310)
ticket:         valid (caps=shell, expires in 23h)
reachability:   ok via relay
                direct-udp: failed (symmetric NAT on remote)
                relay:      ok via https://relay.onyx.net (187ms)
local view:     connected (2 active streams, rx=4.2MB tx=1.1MB)
                connection up: 0:47:12
remote view:    (requires v0.3.4 meta:info RPC)
```

v0.3.2 ships everything except `remote view`, which shows a
placeholder. v0.3.4 delivers `meta:info` RPC + peer pair protocol
together (they share new ALPN territory).

Exit code: 0 if any reachability path succeeded; 1 if all failed.

### 3.7. `portl peer ls --active`

Existing `peer ls` shows the static peer store. `--active`
overlays runtime connection state from the agent IPC.

```text
$ portl peer ls --active
LABEL       ENDPOINT        REL       ORIGIN    PATH         RTT     UP
self        bba96591…b265   self      self      —            —       —
max         bba96591…b265   mutual    paired    direct-udp   23ms    2:14:03
onyx        d65f9e65…7310   mutual    paired    relay        187ms   0:47:12
container   f28fb8aa…0c73   mutual    raw       —            —       not connected
```

When the agent is unreachable, `--active` degrades gracefully: per-
peer PATH / RTT / UP columns show `—`, with a status line at top
of output:

```text
(agent unreachable; showing static peer store only)
```

`portl status peers` is a thin alias for `portl peer ls --active`.

## 5. Implementation plan

### 5.1. Module layout

```
crates/portl-cli/src/commands/status/

  dashboard.rs          # composite view (all sections)
  identity.rs           # identity section
  agent.rs              # agent section
  service.rs            # service section
  peers.rs              # peers section (delegates to `peer ls --active`)
  tickets.rs            # tickets section (delegates to `ticket ls`)

```

### 5.2. Agent IPC surface

The metrics socket currently serves OpenMetrics text on any
request. v0.3.2 adds minimal HTTP path routing:

| Path | Response |
|---|---|
| `GET /` or `GET /metrics` | OpenMetrics text (existing behavior) |
| `GET /status` | JSON snapshot (new) |
| `GET /status/connections` | JSON connection list (new) |
| `GET /status/network` | JSON network info (new) |

JSON shape example (`GET /status`):

```json
{
  "schema": 1,
  "agent": {
    "pid": 1234,
    "started_at": "2025-11-12T09:28:14Z",
    "home": "/Users/thinh/.local/share/portl",
    "version": "0.3.2"
  },
  "connections": [
    {
      "peer_eid": "bba9...b265",
      "label": "max",
      "path": "direct-udp",
      "rtt_ms": 23,
      "bytes_rx": 4404019,
      "bytes_tx": 1153024,
      "up_since": "2025-11-12T09:28:23Z"
    }
  ],
  "network": {
    "home_relay": null,
    "nat_type": "cone",
    "discovery": { "dns": true, "pkarr": true, "mdns": true }
  }
}
```

Request parsing is minimal: read until `\r\n\r\n`, match path,
dispatch. No full HTTP stack; ~80 LOC of parser logic.

### 5.3. Connection tracking in agent

Agent currently tracks `active_connections` as a gauge. v0.3.2
adds a richer registry keyed by endpoint_id:

```rust
pub struct ConnectionRegistry {
    conns: DashMap<EndpointId, ConnectionHandle>,
}

pub struct ConnectionHandle {
    pub peer_eid: EndpointId,
    pub started_at: Instant,
    pub path_kind: PathKind,      // Direct | Relay | Mixed
    pub rtt: AtomicU64,           // microseconds
    pub bytes_rx: AtomicU64,
    pub bytes_tx: AtomicU64,
}

pub enum PathKind { DirectUdp, Relay, Mixed }
```

Populated from iroh's `Endpoint::conn_type` + `connection_stats()`.
RTT updated on each `remote_address_changed` event. Byte counters
incremented in the existing read/write hot paths.

### 5.4. `--watch` implementation

Lives in `status/mod.rs`:

```rust
async fn run_watch(args: StatusArgs, interval: Duration) -> Result<()> {
    let mut signal = tokio::signal::ctrl_c();
    loop {
        print!("\x1b[2J\x1b[H");
        if let Err(e) = render(&args).await {
            eprintln!("(status unavailable: {e}) — retrying in {:?}", interval);
        }
        tokio::select! {
            _ = tokio::time::sleep(interval) => {}
            _ = &mut signal => break,
        }
    }
    Ok(())
}
```

No extra crates; uses existing `tokio::signal` and `tokio::time`.

### 5.5. JSON schema crate organization

Serde structs live in `crates/portl-cli/src/commands/status/json.rs`
and are feature-gated behind `--features json` to keep the default
build lean? **No** — serde is already a workspace dep (tickets
+ peers); no gating needed. Structs are `pub(crate)` and
re-exported under `commands::status::json` for tests.

### 5.6. `portl peer ls --active` implementation

Extends existing `peer ls`:

```rust
pub struct LsArgs {
    pub active: bool,
    pub json: bool,
}

async fn run_ls(args: LsArgs) -> Result<()> {
    let store = peer_store::load()?;
    let overlay = if args.active {
        agent_ipc::fetch_connections().await.ok()  // may be None
    } else {
        None
    };
    render(store, overlay, args.json)
}
```

Gracefully handles `overlay=None` (agent down → dashes).

## 6. Testing plan

- JSON schema v1 structs round-trip through serde.
- HTTP route dispatcher in agent handles `/`, `/metrics`,
  `/status`, `/status/connections`, `/status/network`, unknown
  paths (404).

- `--watch --json` rejected at arg-parse time.

### 6.2. Integration tests

- Boot agent, connect a peer, hit `/status` via direct UDS dial,
  verify connection appears in JSON snapshot.
- `portl status --json | jq` — schema 1 validates via JSON-schema
  file committed under `crates/portl-cli/tests/schema/status-v1.json`.
- `portl peer ls --active` when agent down: falls back to static
  view, prints fallback notice, exits 0.
- `portl status onyx` when peer reachable: exit 0, reachability
  fields populated.
- `portl status onyx` when peer unreachable: exit 1, failure
  reasons listed.
- `--watch` runs 3 iterations then Ctrl+C cleanly (simulated via
  `tokio::signal` injection).

## 7. Migration / back-compat

### 7.1. CLI surface

Breaking changes: **one**. `portl doctor` default output shrinks —
passing checks are now hidden unless `--verbose` is passed. Scripts
that grep for specific "ok" lines will break. Alternative
available: `--verbose` restores prior behavior.

Rationale: `doctor` is the alarm verb. It shouldn't chatter. Users
who want the full scan for diagnostic runs pass `--verbose`.

### 7.2. Agent IPC

Additive. `GET /` still returns OpenMetrics text (Prometheus scrape
compat preserved). New routes under `/status` are strictly new.

### 7.3. Config + env

No new env vars. Agent metrics socket path unchanged.

### 7.4. JSON schema evolution

`schema: 1` is the commitment. Additive (new fields) is allowed
without version bump. Removals or type-changes bump to `schema: 2`.
Consumers should treat unknown fields as opaque.

## 8. Security considerations

### 8.1. IPC surface

`metrics.sock` is already mode 0600; only the agent's user can
read. New `/status` routes inherit that ACL. No additional
exposure.

### 8.2. JSON output on terminals

When `--json` is combined with stdout redirection into files on
shared systems, the same caveats as any other sensitive-config
file apply. Endpoint IDs and peer labels are not secrets (they
appear in `peers.json` already mode 0600). No additional hardening
needed.

### 8.3. `portl status <target>` info leak

The probe dials the target with the caller's identity. Target sees
a connection attempt. Targets that don't want to be probed
shouldn't have tickets/peers pointing at them — same as any other
connection.

No remote-side data leaks; remote-view is a placeholder in v0.3.2.
v0.3.4 `meta:info` RPC will respect cap boundaries (only callers
with `meta:info` cap get remote view).

## 9. Open questions

1. **Color in `--json` stderr errors**: current ANSI escapes work
   fine for terminal users. Strip in JSON mode to be safe.
2. **`portl status --short`**: one-line status for tmux status
   bars. Defer to v0.3.3 if demand surfaces; `watch -n 5 'portl
   status identity'` covers the common case.
3. **Connection registry memory**: bounded by unique peers ever
   seen since agent boot. In practice <100 entries; no pressure.
   Can add LRU cap if needed later.
4. **IPC protocol choice**: HTTP/1.1 text over UDS vs. newline-
   delimited JSON-RPC. Going with HTTP — trivial to curl for
   debug, matches existing OpenMetrics scrape pattern.
5. **`--watch` clear-screen behavior on non-TTY**: detect via
   `is_terminal()`; if not a TTY, emit a `--- tick T ---` separator
   instead of clear. Useful for piping into `tee` for session
   logging.

## 11. Relationship to later releases

- **v0.3.3 relay** gains a `portl status relay` section (spec
  170 §4.2). Agent IPC `/status/relay` route added in v0.3.3;
  v0.3.2 doesn't stub it.
- **v0.3.4 peer pairing** uses `portl peer ls --active` to verify
  a freshly paired peer is reachable; adds `origin=paired|
  accepted` columns.
- **v0.3.4 `meta:info`** fills in the remote view of `portl status
  <target>`.
- **v0.4 `--follow`** adds a new IPC channel (pub-sub over UDS or
  separate socket) for streaming events. `status` gains
  `--follow` alongside `--watch`; watch = polled, follow =
  pushed.
