# 150 — v0.1.1 Safety Net

> Status: **shipped in `v0.1.1`**. First release in
> the three-release sequence that lands the contracts in
> [`140-v0.2-operability.md`](140-v0.2-operability.md).
> Implementation recipe retired per `../plans/README.md` policy;
> see [`../../CHANGELOG.md`](../../CHANGELOG.md) for the final
> shipped scope.
>
> Implementation deviations from the original recipe, documented in commit
> bodies and in §7 below:
>
> - Test harness inlined as `crates/portl-agent/tests/common/mod.rs`
>   (AuditCapture, per-binary watchdog) rather than a `test_support`
>   module on the library crate.
> - Reject reasons dispatch on a dedicated `SpawnReject` enum local
>   to `shell_handler`, not on `ShellReason`. Wire strings unchanged.
> - `audit_session_id` threaded as a function argument into the
>   wait-for-child task rather than a `ShellProcess` field.
> - `panic_abort` test shells out to `cargo build --release --bin
>   portl-agent --features test-panic-trigger` directly rather than
>   via a `test_bin` shim.
>
> Two review-driven hardening items also landed on top of the base
> implementation and are reflected in the contracts below:
> - PTY master fd is marked `FD_CLOEXEC` before `Command::spawn` so
>   the forked child cannot inherit it (fixes a subtle hangup-
>   semantics bug that would let the slave side miss SIGHUP when
>   the parent closed master).
> - `shell_start` / `shell_exit` / `shell_reject` field names and
>   payloads match §3.2 exactly: `pid`, `mode`, `exit_code`,
>   `duration_ms` are present; field names are `ticket_id` and
>   `caller_endpoint_id`, not `…_hex`.
>
> Full acceptance (gates 1–13 in §4) runs at release-cut time on
> `max` (darwin-aarch64) and `vn3` (linux-x86_64).

## 1. Summary

v0.1.1 is a non-breaking patch that ships the three Part C items
from `140 §13` that apply cleanly to the v0.1 runtime without
touching any CLI, config, or filesystem surface. It gives existing
v0.1.0 operators the most valuable hardening — bounded resources
on spawn, balanced audit, and supervisor-observable crashes —
without waiting on v0.2.0's breaking changes.

Scope:

- **§13.1 rlimits on every spawn path** (Linux gate enforced; macOS
  best-effort).
- **§13.5 balanced audit** (every accepted session emits exactly one
  `shell_start` + one `shell_exit`; rejections emit `shell_reject`).
- **§13.6 `panic = "abort"` + panic-site audit** (release profile
  flag + pre-flip audit of latent `.expect(...)` panics).

Out of scope for v0.1.1: anything else in `140`.

## 2. Why these three

### 2.1 Chosen for ship-ahead-of-v0.2

Each item here:

1. Is **non-breaking**: no CLI change, no config-format change, no
   on-disk layout change, no new env-var required.
2. Has **immediate operational value**: operators running v0.1.0 on
   real hosts benefit the day v0.1.1 lands.
3. Is **low-risk**: tightly scoped, mechanically verifiable, easy to
   revert. None of them blocks or interacts with the rest of Part C.
4. **De-risks v0.2** by shaking out platform-specific gotchas
   (especially `pre_exec` behaviour on Darwin) in a small release.

### 2.2 Why these three specifically, from the Part C list

| Item | Shippable without v0.2 prerequisites? | Why |
| --- | --- | --- |
| §13.1 rlimits | yes | `pre_exec` hook already exists on exec path; extends the existing hook + adds one on the PTY path. |
| §13.2 pgroup escalator | no | Depends on §13.4 cancellation wiring to be genuinely useful. Landing alone doesn't buy much over the existing `SIGHUP` behaviour. |
| §13.3 PTY drain | no | Requires thread-to-async refactor; risk is disproportionate to v0.1.1's "small and safe" charter. |
| §13.4 revocation kills live | no | Needs §13.2 escalator + session registry changes; can't ship in isolation. |
| §13.5 audit balance | yes | Additive: emit `shell_reject` for rejections, `shell_start` for accepted sessions. No behaviour change. |
| §13.6 panic=abort | yes | Cargo profile flag; panic-site audit is read-only; unit templates updated. No code-path changes. |
| §13.7 slow-task detection | no | Depends on a new `portl_core::runtime::slow_task` helper + metrics plumbing that v0.2 introduces. |
| §13.8 revocations ceiling | no | New env var (`PORTL_REVOCATIONS_MAX_BYTES`); v0.1 env surface is frozen. |
| §13.9 graceful shutdown | no | Requires §13.2 escalator + session registry walk. |

### 2.3 What v0.1.1 deliberately *does not* do

- Does not change `shell_handler.rs`'s single-SIGHUP termination
  (§13.2 lands in v0.2).
- Does not touch the alias store; `rusqlite` stays in v0.1.1 and is
  isolated in v0.1.2 (see `160-v0.1.2-alias-isolation.md`).
- Does not change how revocations interact with live sessions.
- Does not add any new env vars or CLI flags.

## 3. Contracts this release establishes

### 3.1 rlimits on every spawn path (§13.1)

Both `portl shell` (PTY) and `portl exec` (non-PTY) spawn their
children under these `setrlimit(2)` values, applied in a `pre_exec`
hook before `execve`:

| Resource | Limit |
| --- | --- |
| `RLIMIT_NOFILE` | `4096` |
| `RLIMIT_NPROC` | `512` (Linux per-uid; macOS per-process) |
| `RLIMIT_CORE` | `0` |
| `RLIMIT_CPU` | `86_400` seconds |
| `RLIMIT_FSIZE` | `10 GiB` |

The values are **not configurable in v0.1.1**; they are baked in.
If operator feedback demands tuning, v0.3 adds env-var overrides.

**Platform caveat.** The fork-bomb acceptance test passes on Linux
and is skipped (marked `#[cfg(target_os = "linux")]`) on macOS. On
Darwin `RLIMIT_NPROC` is per-process; a fork-bomb inside
`portl shell` on macOS is bounded by the shell's own process
budget, not the agent's uid budget. macOS gets the other four
limits (`NOFILE`, `CORE`, `CPU`, `FSIZE`). This is documented in
CHANGELOG as a known platform limitation.

**Implementation shape.** v0.1 had a `pre_exec` hook only on the
non-PTY exec path. v0.1.1 adds an equivalent hook on the PTY path
by replacing the `portable_pty::CommandBuilder` spawn with a
direct `CommandExt::pre_exec` implementation that does
`openpty(3)` + `setsid` + `ioctl(TIOCSCTTY)` + `dup2(slave, 0|1|2)`
+ `setrlimit(…)` inside the hook. `portable_pty` is dropped from
the agent crate entirely.

The PTY master fd is marked `FD_CLOEXEC` (via `fcntl(F_SETFD,
FD_CLOEXEC)`) in the parent **before** `Command::spawn`. This
prevents the forked child from inheriting an open master handle,
which would otherwise keep the master's open-count > 0 from the
child side and suppress the slave-side `SIGHUP` / EOF when the
parent closes its master. The slave fd is deliberately not
`CLOEXEC`: `dup2` into 0/1/2 inside the `pre_exec` hook clears
`FD_CLOEXEC` on the dup targets, which is what we want.

Code references (current):
- `crates/portl-agent/src/shell_handler.rs` — `spawn_pty_blocking`
  (PTY path, `pre_exec` closure with `FD_CLOEXEC` on master,
  `setsid`, `TIOCSCTTY`, `dup2`, `apply_rlimits`)
- `crates/portl-agent/src/shell_handler.rs` —
  `install_exec_rlimits_pre_exec` (non-PTY exec path)
- `crates/portl-agent/src/shell_handler.rs` — `apply_rlimits`
  (the shared rlimit block; `NPROC` is `#[cfg(target_os = "linux")]`-gated)

### 3.2 Balanced audit (§13.5)

**Invariant**: every accepted session emits exactly one
`audit.shell_start` record followed by exactly one
`audit.shell_exit`. Rejections emit one `audit.shell_reject`
record with a `reason` field and do **not** emit `shell_start` /
`shell_exit`.

Fields:

| Record | Fields |
| --- | --- |
| `shell_start` | `ticket_id`, `caller_endpoint_id`, `session_id`, `pid`, `mode` (`pty`\|`exec`), `shell_user`, `shell_argv0` |
| `shell_exit` | `ticket_id`, `caller_endpoint_id`, `session_id`, `pid`, `exit_code`, `duration_ms` |
| `shell_reject` | `ticket_id`, `caller_endpoint_id`, `reason` (`argv_empty` \| `caps_denied` \| `path_probe_failed` \| `pty_allocation_failed` \| `uid_lookup_failed` \| `user_switch_refused`) |

Reason strings are dispatched from a typed `SpawnReject` enum
local to `shell_handler`, not from request-shape heuristics. A
`pre_exec`-time failure (rlimit EPERM, TIOCSCTTY EPERM) surfaces
as `path_probe_failed` or `pty_allocation_failed` as appropriate;
future failure modes require a new enum variant + a new wire
string + an update to this table.

v0.1.0 emitted `shell_spawn` and `shell_exit` but not
`shell_reject`. v0.1.1 renames `shell_spawn` to `shell_start`,
adds the `session_id` field (UUIDv4, generated at accept time),
and adds `pid` + `mode` to `shell_start`, plus `exit_code` and
`duration_ms` to `shell_exit`. The rename is a hard cutover — no
dual emission during the upgrade window, no field-name aliases.
Audit JSONL is pre-1.0, the schema is unstable, and no external
consumer exists; CHANGELOG notes the break.

Every accepted session (one `shell_start`) emits exactly one
matching `shell_exit` with the same `session_id`. The
`tests/audit_balance.rs` integration test verifies this
invariant; `tests/audit_reject.rs` verifies that rejected
sessions emit `shell_reject` with the correct `reason` string
and never a `shell_start` / `shell_exit`.

### 3.3 `panic = "abort"` + panic-site audit (§13.6)

`[profile.release]` sets `panic = "abort"`. Any panic on any async
task terminates the process via `SIGABRT`. The supervisor
restarts.

Prerequisite work landed in commit `06f62d2` (panic-site audit):

1. Every `expect` / `unwrap` / `panic!` call site in `crates/` was
   triaged into {infallible / documented, recoverable, test-only}.
   Raw hit count 153; 148 test-only, 3 infallible (each with a
   `// SAFETY(panic): …` comment), 2 recoverable (rewritten).
2. The two recoverable sites were `RwLock` poison handlers:
   - `crates/portl-agent/src/meta_handler.rs` — poison now
     returns `MetaResp::Error(InternalError)` instead of
     `.expect(...)`.
   - `crates/portl-agent/src/ticket_handler.rs` — poison now
     propagates an `anyhow!` error up `serve_connection`.
3. Full audit scratch file at
   `scratch/2026-04-21-v0.1.1-panic-audit.md` (gitignored by
   design; re-run `rg -n 'expect|unwrap|panic!' crates/` to
   regenerate).

The release-build panic test lives at
`crates/portl-agent/tests/panic_abort.rs`. It builds `portl-agent`
with the `test-panic-trigger` feature enabled, sets
`PORTL_TEST_PANIC_AT=startup`, runs the binary, and asserts the
exit status is `SIGABRT` (or equivalent: exit code 134, or signal
None with abort semantics). A non-zero unwind exit would fail the
assertion — verifying that `panic = "abort"` is actually what's
firing, not unwinding that happens to return `Err`.

The `test-panic-trigger` feature is opt-in, not in `default`, and
is not referenced by the release-build job in
`.github/workflows/ci.yml`, so it cannot leak into shipped tarballs.

Supervisor-side: `portl install` does not exist yet in v0.1.1.
Operators who run v0.1.1 as a systemd/launchd service are
responsible for their own unit's `Restart=on-failure`. Example
units will ship in v0.2 when `portl install` lands (see 140 §7).

## 4. Acceptance gates

A v0.1.1 build is releasable when all of the following hold. Each
gate is a script in `scripts/acceptance/v0.1.1/` or a test in the
workspace; they run in CI on ubuntu-latest and macos-latest.

### 4.1 Runtime behaviour

1. **(Linux)** `portl shell <ticket>` → run `:(){ :|:& };:` → the
   fork-bomb is killed by `RLIMIT_NPROC`; agent's uid process
   count returns to baseline within 5 s; agent itself stays
   responsive to a second `portl status` call within 2 s.
2. **(both)** `portl exec <ticket> -- bash -c 'ulimit -n'` prints
   `4096`.
3. **(both)** `portl exec <ticket> -- bash -c 'ulimit -c'` prints
   `0`.
4. **(Linux)** `prlimit --pid <grandchild-pid>` shows `NPROC=512`,
   `NOFILE=4096`, verifying rlimit inheritance past the immediate
   shell.

### 4.2 Audit balance

5. `portl exec <ticket> -- true`, grep the audit JSONL: exactly one
   `shell_start` and one `shell_exit` record, matching
   `session_id`s.
6. `portl shell <ticket>` with an invalid `--user` → agent's audit
   JSONL contains one `shell_reject` with
   `reason=user_switch_refused`; no `shell_start` or `shell_exit`.
7. `kill -9 $(pidof portl-agent)` mid-session, restart, inspect the
   audit JSONL for the killed session: exactly one `shell_start`;
   `shell_exit` may be absent if the kill preceded the exit flush
   (this is acceptable in v0.1.1; v0.2 §13.9 promises durability).

### 4.3 Panic behaviour

8. Induced panic (a cfg-gated test-only trigger inside the shell
   handler) under release build: agent process exits non-zero
   within 100 ms; no half-dead tokio runtime state.
9. Panic-site audit: `grep -rn '\.expect(\|\.unwrap(\)' crates/`
   across the workspace returns only sites that are either (a)
   triaged-and-commented or (b) test-only.

### 4.4 No regressions

10. Full `cargo test --workspace --all-features` passes on both
    platforms. Count matches v0.1.0 + new tests only; no removals.
11. `cargo deny check` clean.
12. `portl doctor` on a v0.1.0 `$PORTL_HOME` succeeds without
    migration.
13. Existing systemd / launchd / Docker deployments running v0.1.0
    survive an upgrade to v0.1.1 without unit-file edits.

## 5. Risks and mitigations

| Risk | Severity | Mitigation |
| --- | --- | --- |
| PTY `pre_exec` replacement breaks on a macOS quirk (PTY setup ordering, `ioctl(TIOCSCTTY)` EPERM under non-root) | HIGH | Landed with `tests/pty_spawn.rs` integration test that spawns a shell through the PTY path and verifies `tty(1)` reports a valid `/dev/pts*` device (or Darwin-equivalent). Test passes on both macos-latest and ubuntu-latest in CI. Revert path is `git revert <PTY-refactor-commit>`; `portable_pty` dep re-add is one cargo line. |
| `panic = "abort"` exposes a panic path nobody noticed in v0.1.0 | MEDIUM | Panic-site audit (§3.3) landed first. Mutex-poison sites reviewed individually; 2 recoverable sites rewritten; 3 infallible sites commented. `tests/panic_abort.rs` verifies the release build aborts, not unwinds. |
| Renaming `shell_spawn` → `shell_start` breaks any audit-log consumers | LOW | No external consumers exist; documented in CHANGELOG; pre-1.0. |
| Field-name rename `ticket_id_hex` → `ticket_id` / `caller_endpoint_id_hex` → `caller_endpoint_id` breaks any audit-log consumers | LOW | Same as above; pre-1.0 JSONL schema, no consumers. |
| Hard-coded rlimits (`NOFILE=4096`) too low for some interactive workloads | LOW | 4096 is deliberately generous; if reported, tunable in v0.3. |
| CI missing a fork-bomb test failure on Darwin because `RLIMIT_NPROC` is per-process | LOW | Test gated with `#[cfg(target_os = "linux")]` in `tests/rlimits.rs`; Darwin caveat documented in CHANGELOG and 140 §13.1. |
| PTY master fd inherited by child, suppressing slave-side SIGHUP | HIGH (caught in M1+M2 roundtable review) | `FD_CLOEXEC` set on master via `fcntl` before `Command::spawn`. Regression test verifies no master fd leaks into the child's fd table. |

## 6. Release mechanics

Tag: `v0.1.1`. Target: commit on `main` that contains the plan's
final "Commit" step.

Artifacts: same four tarballs as v0.1.0 (musl x86_64 + aarch64,
darwin x86_64 + aarch64), same packaging (`zstd -19`), same
sha256 sidecars.

platform caveat. Release notes point at this spec file for detail.

once on `max ↔ vn3` (scenarios A, B, C from the M7 close-out), on
v0.1.1 tarballs freshly downloaded. No behaviour changes expected
beyond the three §3 contracts.
