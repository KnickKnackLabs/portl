# 150 — v0.1.1 Safety Net

> Status: **proposed**. First release in the three-release sequence
> that lands the contracts in [`140-v0.2-operability.md`](140-v0.2-operability.md).
> Companion plan: [`../plans/010-v0.1.1-safety-net.md`](../plans/010-v0.1.1-safety-net.md).

## 1. Summary

v0.1.1 is a non-breaking patch that ships the three Part C items
from `140 §13` that apply cleanly to the v0.1 runtime without
touching any CLI, config, or filesystem surface. It gives existing
v0.1.0 operators the most valuable hardening — bounded resources
on spawn, balanced audit, and supervisor-observable crashes —
without waiting on v0.2.0's breaking changes.

Scope:

- **§13.1 rlimits on every spawn path** (Linux gate enforced; macOS
  best-effort).
- **§13.5 balanced audit** (every accepted session emits exactly one
  `shell_start` + one `shell_exit`; rejections emit `shell_reject`).
- **§13.6 `panic = "abort"` + panic-site audit** (release profile
  flag + pre-flip audit of latent `.expect(...)` panics).

Out of scope for v0.1.1: anything else in `140`.

## 2. Why these three

### 2.1 Chosen for ship-ahead-of-v0.2

Each item here:

1. Is **non-breaking**: no CLI change, no config-format change, no
   on-disk layout change, no new env-var required.
2. Has **immediate operational value**: operators running v0.1.0 on
   real hosts benefit the day v0.1.1 lands.
3. Is **low-risk**: tightly scoped, mechanically verifiable, easy to
   revert. None of them blocks or interacts with the rest of Part C.
4. **De-risks v0.2** by shaking out platform-specific gotchas
   (especially `pre_exec` behaviour on Darwin) in a small release.

### 2.2 Why these three specifically, from the Part C list

| Item | Shippable without v0.2 prerequisites? | Why |
| --- | --- | --- |
| §13.1 rlimits | yes | `pre_exec` hook already exists on exec path; extends the existing hook + adds one on the PTY path. |
| §13.2 pgroup escalator | no | Depends on §13.4 cancellation wiring to be genuinely useful. Landing alone doesn't buy much over the existing `SIGHUP` behaviour. |
| §13.3 PTY drain | no | Requires thread-to-async refactor; risk is disproportionate to v0.1.1's "small and safe" charter. |
| §13.4 revocation kills live | no | Needs §13.2 escalator + session registry changes; can't ship in isolation. |
| §13.5 audit balance | yes | Additive: emit `shell_reject` for rejections, `shell_start` for accepted sessions. No behaviour change. |
| §13.6 panic=abort | yes | Cargo profile flag; panic-site audit is read-only; unit templates updated. No code-path changes. |
| §13.7 slow-task detection | no | Depends on a new `portl_core::runtime::slow_task` helper + metrics plumbing that v0.2 introduces. |
| §13.8 revocations ceiling | no | New env var (`PORTL_REVOCATIONS_MAX_BYTES`); v0.1 env surface is frozen. |
| §13.9 graceful shutdown | no | Requires §13.2 escalator + session registry walk. |

### 2.3 What v0.1.1 deliberately *does not* do

- Does not change `shell_handler.rs`'s single-SIGHUP termination
  (§13.2 lands in v0.2).
- Does not touch the alias store; `rusqlite` stays in v0.1.1 and is
  isolated in v0.1.2 (see `160-v0.1.2-alias-isolation.md`).
- Does not change how revocations interact with live sessions.
- Does not add any new env vars or CLI flags.

## 3. Contracts this release establishes

### 3.1 rlimits on every spawn path (§13.1)

Both `portl shell` (PTY) and `portl exec` (non-PTY) spawn their
children under these `setrlimit(2)` values, applied in a `pre_exec`
hook before `execve`:

| Resource | Limit |
| --- | --- |
| `RLIMIT_NOFILE` | `4096` |
| `RLIMIT_NPROC` | `512` (Linux per-uid; macOS per-process) |
| `RLIMIT_CORE` | `0` |
| `RLIMIT_CPU` | `86_400` seconds |
| `RLIMIT_FSIZE` | `10 GiB` |

The values are **not configurable in v0.1.1**; they are baked in.
If operator feedback demands tuning, v0.3 adds env-var overrides.

**Platform caveat.** The fork-bomb acceptance test passes on Linux
and is skipped (marked `#[cfg(target_os = "linux")]`) on macOS. On
Darwin `RLIMIT_NPROC` is per-process; a fork-bomb inside
`portl shell` on macOS is bounded by the shell's own process
budget, not the agent's uid budget. macOS gets the other four
limits (`NOFILE`, `CORE`, `CPU`, `FSIZE`). This is documented in
CHANGELOG as a known platform limitation.

**Implementation shape.** v0.1 has a `pre_exec` hook only on the
non-PTY exec path (`shell_handler.rs:327, 785-804`). v0.1.1 adds
an equivalent hook on the PTY path by replacing the
`portable_pty::CommandBuilder` spawn with a direct
`CommandExt::pre_exec` implementation that does `openpty(3)` +
`setsid` + `ioctl(TIOCSCTTY)` + `setrlimit(…)` inside the hook.

### 3.2 Balanced audit (§13.5)

**Invariant**: every accepted session emits exactly one
`audit.shell_start` record followed by exactly one
`audit.shell_exit`. Rejections emit one `audit.shell_reject`
record with a `reason` field and do **not** emit `shell_start` /
`shell_exit`.

Fields:

| Record | Fields |
| --- | --- |
| `shell_start` | `ticket_id`, `caller_endpoint_id`, `session_id`, `pid`, `mode` (`pty`\|`exec`), `shell_user`, `shell_argv0` |
| `shell_exit` | `ticket_id`, `caller_endpoint_id`, `session_id`, `pid`, `exit_code`, `duration_ms` |
| `shell_reject` | `ticket_id`, `caller_endpoint_id`, `reason` (`path_probe_failed` \| `uid_lookup_failed` \| `user_switch_refused` \| `pty_allocation_failed` \| other) |

v0.1.0 emits `shell_spawn` and `shell_exit` but not `shell_reject`.
v0.1.1 renames `shell_spawn` to `shell_start` and adds the
`session_id` field (UUIDv4, generated at accept time). Consumers
reading the JSONL audit log will see both field names during the
v0.1.0→v0.1.1 upgrade window; no migration is required.

### 3.3 `panic = "abort"` + panic-site audit (§13.6)

`[profile.release]` adds `panic = "abort"`. Any panic on any async
task terminates the process non-zero. The supervisor restarts.

Prerequisite work, done as part of this release's plan:

1. `grep -rn 'expect\|unwrap' crates/` across the workspace; every
   hit is triaged into "caller-tolerated / fatal-by-design /
   needs-graceful-return". Triage results are committed as
   code comments next to each site.
2. Mutex-poison sites that currently `anyhow!` are reviewed: if the
   poison is actually recoverable, the code continues with a
   `warn!`; if not, it panics explicitly with a useful message.
3. Any site returning `.expect(...)` on user input is replaced with
   a proper `Result`.

After that review lands, `panic = "abort"` is enabled and v0.1.1
ships.

Supervisor-side: `portl install` does not exist yet in v0.1.1.
Operators who run v0.1.1 as a systemd/launchd service are
responsible for their own unit's `Restart=on-failure`. Example
units are shipped in `docs/specs/080-cli.md` and `README.md`.

## 4. Acceptance gates

A v0.1.1 build is releasable when all of the following hold. Each
gate is a script in `scripts/acceptance/v0.1.1/` or a test in the
workspace; they run in CI on ubuntu-latest and macos-latest.

### 4.1 Runtime behaviour

1. **(Linux)** `portl shell <ticket>` → run `:(){ :|:& };:` → the
   fork-bomb is killed by `RLIMIT_NPROC`; agent's uid process
   count returns to baseline within 5 s; agent itself stays
   responsive to a second `portl status` call within 2 s.
2. **(both)** `portl exec <ticket> -- bash -c 'ulimit -n'` prints
   `4096`.
3. **(both)** `portl exec <ticket> -- bash -c 'ulimit -c'` prints
   `0`.
4. **(Linux)** `prlimit --pid <grandchild-pid>` shows `NPROC=512`,
   `NOFILE=4096`, verifying rlimit inheritance past the immediate
   shell.

### 4.2 Audit balance

5. `portl exec <ticket> -- true`, grep the audit JSONL: exactly one
   `shell_start` and one `shell_exit` record, matching
   `session_id`s.
6. `portl shell <ticket>` with an invalid `--user` → agent's audit
   JSONL contains one `shell_reject` with
   `reason=user_switch_refused`; no `shell_start` or `shell_exit`.
7. `kill -9 $(pidof portl-agent)` mid-session, restart, inspect the
   audit JSONL for the killed session: exactly one `shell_start`;
   `shell_exit` may be absent if the kill preceded the exit flush
   (this is acceptable in v0.1.1; v0.2 §13.9 promises durability).

### 4.3 Panic behaviour

8. Induced panic (a cfg-gated test-only trigger inside the shell
   handler) under release build: agent process exits non-zero
   within 100 ms; no half-dead tokio runtime state.
9. Panic-site audit: `grep -rn '\.expect(\|\.unwrap(\)' crates/`
   across the workspace returns only sites that are either (a)
   triaged-and-commented or (b) test-only.

### 4.4 No regressions

10. Full `cargo test --workspace --all-features` passes on both
    platforms. Count matches v0.1.0 + new tests only; no removals.
11. `cargo deny check` clean.
12. `portl doctor` on a v0.1.0 `$PORTL_HOME` succeeds without
    migration.
13. Existing systemd / launchd / Docker deployments running v0.1.0
    survive an upgrade to v0.1.1 without unit-file edits.

## 5. Risks and mitigations

| Risk | Severity | Mitigation |
| --- | --- | --- |
| PTY `pre_exec` replacement breaks on a macOS quirk (PTY setup ordering, `ioctl(TIOCSCTTY)` EPERM under non-root) | HIGH | Ship to macos-latest CI first; keep the v0.1.0 `portable_pty` path behind a `#[cfg]` feature flag we can revert to. |
| `panic = "abort"` exposes a panic path nobody noticed in v0.1.0 | MEDIUM | Panic-site audit (§3.3) precedes the flag flip. Mutex-poison sites reviewed individually. |
| Renaming `shell_spawn` → `shell_start` breaks any audit-log consumers | LOW | No external consumers exist; this is documented in CHANGELOG and is pre-1.0. |
| Hard-coded rlimits (`NOFILE=4096`) too low for some interactive workloads | LOW | 4096 is deliberately generous; if reported, tunable in v0.3. |
| CI missing a fork-bomb test failure on Darwin because `RLIMIT_NPROC` is per-process | LOW | Gate test with `#[cfg(target_os = "linux")]`; skip on macOS with a documented `SKIP` reason. |

## 6. Release mechanics

Tag: `v0.1.1`. Target: commit on `main` that contains the plan's
final "Commit" step.

Artifacts: same four tarballs as v0.1.0 (musl x86_64 + aarch64,
darwin x86_64 + aarch64), same packaging (`zstd -19`), same
sha256 sidecars.

CHANGELOG: entry for v0.1.1 summarising `§3` contracts + the
platform caveat. Release notes point at this spec file for detail.

Post-release verification: the cross-machine smoke scenarios run
once on `max ↔ vn3` (scenarios A, B, C from the M7 close-out), on
v0.1.1 tarballs freshly downloaded. No behaviour changes expected
beyond the three §3 contracts.
