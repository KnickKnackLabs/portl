# 180 — v0.3.4 Peer Pairing: Network Handshake + Relay Hints

> Status: **draft**, targeting v0.3.4. Depends on v0.3.3 relay
> release (for relay-hint consumers and the closed-policy relay
> that pair-protocol peers will use). Completes the v0.3.0 peer
> vocabulary that deferred the network handshake surface.

## 1. Summary

v0.3.4 adds the three-verb peer pair protocol deferred from v0.3.0:
`portl peer invite`, `portl peer pair`, and `portl peer accept`.
Replaces the current "copy-paste endpoint_id + `portl peer
add-unsafe-raw --mutual --yes` on both sides" workflow with a
proper handshake that mutates both sides' peer stores atomically
under signed nonces.

In the same release, peer entries gain an optional `relay_hint`
field so paired peers can automatically learn each other's preferred
relay. The pair protocol itself carries the hint, so the user never
touches a relay URL after `portl relay serve` + `portl peer pair`
on the other end.

Two shipping outcomes:

1. Three new CLI verbs — `peer invite`, `peer pair <code>`,
   `peer accept <code>` — all backed by a new
   `portl/pair/v1` ALPN.
2. Peer entries v2 carry `relay_hint`; `portl peer ls` surfaces it;
   agents honor it when initiating connections.

Explicit non-goals:

- `peer hold` / `peer resume` temporary suspension — deferred to
  v0.4.
- `peer rename` — deferred to v0.4. Manual rename via edit of
  peers.json continues to work.
- Multi-party invites (one-to-many pair). Each pair is a two-party
  handshake.
- Transitive trust (A pairs with B, B pairs with C ⇒ A trusts C).
  Not happening. Out of scope for portl's trust model.

## 2. Motivation

v0.3.0's `peer add-unsafe-raw` is the "escape hatch" for self-host:
operators copy-paste 64-char hex endpoint_ids between machines,
retype to confirm, and run the command on both sides. Works but
has friction + foot-gun:

- 64-char hex is error-prone to transcribe (esp. over voice).
- Retype-to-confirm protects one side but not the other — the
  inviter still types the friend's eid in, and any typo there
  trusts someone else.
- No atomicity: if one side succeeds and the other fails, the two
  peer stores disagree silently.
- No TTL on the invite; a posted eid lingers forever.
- No way to convey relay-hint or other metadata at pair time.

Peer pair protocol fixes these:

- Base32-encoded invite codes (~80 chars) with embedded TTL and
  authentication nonce.
- Cryptographically verified identity exchange (each side proves
  possession of its signing key via the TLS handshake + a fresh
  signed nonce).
- Atomic bilateral write: both peer stores end up in the expected
  state, or both roll back.
- Natural carrier for future metadata (relay hints, labels, etc.)
  without wire changes.

## 3. Design decisions

### 3.1. Three verbs, not one flag

Explicitly:

```bash
portl peer invite [--ttl 1h] [--for <label-hint>]
    # On machine A. Generates a short-lived invite code.

portl peer pair <code>
    # On machine B. Dials A with the code; both sides
    # end up with mutual peer entries.

portl peer accept <code>
    # On machine B. Dials A with the code; A ends up with
    # outbound-only peer entry for B; B ends up with
    # inbound-only peer entry for A.
```

Rejected: single `portl peer pair <code> [--mutual|--accept]` with
a flag. Reason: mutual-vs-one-way is a security-semantic decision
that deserves a distinct verb, not a flag. Matches the reasoning
from v0.3.0's `peer add-unsafe-raw`: security-critical decisions
live at the verb, not in an option.

Use cases:

- `pair` (mutual): home machines, trusted collaborators, homelab.
  Both sides can initiate connections.
- `accept` (one-way): giving someone ssh-like access into your box
  without getting access back out. Common for customer remote-
  support, production-from-ops-machine, IoT device reach-in.

### 3.2. Invite code structure

Base32-encoded binary payload, ~80 chars:

```text
<version:1 byte=1>
<inviter_endpoint_id:32 bytes>
<nonce:16 bytes>                ; random; used once
<not_after:8 bytes u64 little-endian unix seconds>
<relay_hint_len:1 byte>
<relay_hint_bytes:variable, 0..255>   ; inviter's preferred relay URL if any
```

Encoded as base32 → copy-pastable, QR-friendly, tolerates typos
better than hex.

Invite codes are **not secrets per se** — anyone who has the code
and can reach the inviter's endpoint can complete a pair. The
operator is expected to share codes over trusted channels (DM,
signed email, chat). The nonce + TTL limit the window.

### 3.3. Wire protocol

New ALPN: `portl/pair/v1`.

Wire messages (postcard-encoded, same as `portl/ticket/v1`):

```rust
#[derive(Serialize, Deserialize)]
pub enum PairMessage {
    /// B → A, carrying the invite code decoded
    PairRequest {
        version: u8,            // 1
        nonce: [u8; 16],        // from invite code
        mode: PairMode,         // Pair | Accept
        caller_relay_hint: Option<String>,  // B's relay, if any
    },
    /// A → B after validating nonce + mutating peer store
    PairResponse {
        result: PairResult,
        responder_relay_hint: Option<String>,  // A's relay, if any
        responder_label: Option<String>,       // A's auto-label for B
    },
}

#[derive(Serialize, Deserialize)]
pub enum PairMode { Pair, Accept }

#[derive(Serialize, Deserialize)]
pub enum PairResult {
    Ok,
    NonceExpired,
    NonceUnknown,
    AlreadyPaired,
    PolicyRejected(String),
}
```

### 3.4. Server-side state

New file: `$PORTL_HOME/pending_invites.json` — list of open invites
the agent is waiting for.

```json
{
  "schema": 1,
  "invites": [
    {
      "nonce_hex": "a1b2c3...",
      "issued_at": 1776890000,
      "not_after": 1776893600,
      "for_label": "onyx"
    }
  ]
}
```

`portl peer invite` appends; agent's `peer_store_reload_task`
already polls `peers.json` — we add an analogous task for
`pending_invites.json`. Agent also prunes expired invites on
reload.

On successful pair:

1. Agent validates nonce exists + not expired.
2. Agent extracts caller's endpoint_id from the QUIC TLS peer cert.
3. Agent mutates `peers.json`: insert caller with the right
   relationship for the pair mode.
4. Agent removes the consumed nonce from `pending_invites.json`.
5. Agent sends `PairResponse::Ok` with own `relay_hint` + label.
6. Caller mutates its own `peers.json` with the responder's
   endpoint_id (from TLS peer cert) and the relay_hint.

Atomicity: both writes happen under `atomic-write` semantics that
`peer_store.save()` already provides (write-tmp + rename). If the
caller crashes between receiving Response and writing its local
store, re-running `peer pair <same_code>` hits `NonceUnknown`
(already consumed) and the caller falls back to `peer add-unsafe-
raw` with the responder's endpoint_id it already learned from
the TLS cert.

### 3.5. Peer entry schema v2

```rust
// Existing v1:
pub struct PeerEntry {
    pub label: String,
    pub endpoint_id_hex: String,
    pub accepts_from_them: bool,
    pub they_accept_from_me: bool,
    pub since: u64,
    pub origin: PeerOrigin,
    pub last_hold_at: Option<u64>,
    pub is_self: bool,
}

// v2 adds:
    pub relay_hint: Option<String>,     // RelayUrl as string
    pub schema_version: u8,             // 2
```

Schema bump is backward-compatible: v1 entries loaded as v2 default
`relay_hint: None, schema_version: 1`. Agent's peer-store writer
writes v2 on next save. Old agents reading v2 files see `relay_hint`
as an unknown field and drop it (serde's default behavior).

### 3.6. Origin gains `Paired`

```rust
pub enum PeerOrigin {
    Zelf,          // from `portl install --apply` / `portl init`
    Paired,        // from peer pair protocol  ← NEW in v0.3.4
    Accepted,      // from peer accept protocol ← NEW in v0.3.4
    Raw,           // from `peer add-unsafe-raw`
}
```

`portl peer ls` gains an `ORIGIN` column that uses these labels so
operators can distinguish "safely paired" from "manually pinned."

### 3.7. Relay hint propagation

When an agent initiates a connection to peer X:

1. Agent first attempts direct / holepunch (via iroh defaults).
2. If direct fails + peer has `relay_hint: Some(url)`, agent
   includes that URL in the relay list for this dial.
3. If peer has no hint, agent uses configured default relays only.

Client-side:

```rust
// crates/portl-cli/src/commands/peer_resolve.rs
fn resolve_endpoint_addr(endpoint: &Endpoint, eid: EndpointId) -> EndpointAddr {
    let hint = peer_store.get_by_endpoint(&eid).and_then(|e| e.relay_hint.clone());
    EndpointAddr::new(eid).with_relay_hint(hint)
}
```

Relay hints are **not** enforced — they're preferences. An operator
can still reach a peer via a different relay if the hinted one is
down.

## 4. CLI surface

### 4.1. Invite

```bash
# On machine A (issuer)
$ portl peer invite --ttl 1h --for onyx
code: PORTLINV-AAVABSKF77KGZ7WQGAIA...X5EZ4DKL   (expires in 1h)
share this code over a trusted channel (DM, signed email, etc.)
pair:   portl peer pair <code>     # mutual trust
accept: portl peer accept <code>   # one-way: they can reach you, not vice-versa

# Check pending invites
$ portl peer invite --list
LABEL_HINT     NONCE          EXPIRES_IN     ORIGIN
onyx           a1b2c3d4…     57m            invite
(issued by --for <label-hint>)

# Revoke a pending invite
$ portl peer invite --revoke <nonce-prefix>
revoked invite a1b2c3d4…
```

### 4.2. Pair (mutual)

```bash
# On machine B (acceptor, wants mutual)
$ portl peer pair PORTLINV-AAVABSKF77KGZ7WQGAIA...X5EZ4DKL
dialing inviter...
handshake ok.
added peer 'max'  (bba96591…b265)  — mutual  (paired)
on the other side: added peer 'onyx' (d65f9e65…7310) — mutual (paired)
```

### 4.3. Accept (one-way)

```bash
# On machine B (acceptor, doesn't want inbound)
$ portl peer accept PORTLINV-AAVABSKF77KGZ7WQGAIA...X5EZ4DKL
dialing inviter...
handshake ok.
added peer 'max' (bba96591…b265) — inbound  (accepted)
    (they can reach you; you cannot initiate back)
on the other side: added peer 'onyx' (d65f9e65…7310) — outbound (accepted)
```

## 5. Implementation plan

### 5.1. Module layout

```
crates/portl-proto/src/
  pair_v1.rs            # NEW — ALPN, message types, wire helpers

crates/portl-core/src/
  peer_store.rs         # schema v2 (backward-compat load of v1)
  pair_store.rs         # NEW — pending_invites.json

crates/portl-agent/src/
  pair_handler.rs       # NEW — portl/pair/v1 ALPN accept loop
  lib.rs                # spawn_pair_handler_task wired in

crates/portl-cli/src/commands/peer/
  invite.rs             # NEW — issue invite, list, revoke
  pair.rs               # NEW — consume invite (mutual)
  accept.rs             # NEW — consume invite (one-way)

crates/portl-cli/src/lib.rs   # new subcommand variants
```

### 5.2. Invite code encoding

Use `data-encoding` crate (already a transitive dep via iroh) for
base32. Fixed-format binary per §3.2 so decoding is trivial and
forward-compatible — version byte gates future expansions.

### 5.3. Nonce lifecycle

- Issued by `peer invite` → written to `pending_invites.json`.
- Consumed on first successful pair/accept handshake.
- Expired on TTL (default 1h, configurable `--ttl`).
- Agent prunes expired on reload.
- Revocable via `peer invite --revoke <nonce-prefix>`.

### 5.4. Collision handling

- Caller's endpoint_id already present in peer_store: respond
  `AlreadyPaired(existing_label)`; no state change. Caller can
  still call `peer rename <new> <old>` (future) or just accept
  the existing entry.
- Label collision between pair-protocol-generated label and
  existing store entry: fall back to auto-generated label
  (first 8 hex chars of endpoint_id), same as `add-unsafe-raw`.

### 5.5. Relay-hint in pair wire

The inviting agent reads its own `RelayServerConfig` (from v0.3.3)
and puts the hostname in the `relay_hint` field of the invite
code and `PairResponse`. The caller does the same symmetrically.
Hints are advisory — receivers may ignore them and use their own
relay config.

### 5.6. Test strategy

- Unit: invite code encode/decode roundtrip (version, nonce, TTL,
  relay-hint).
- Unit: peer entry v1 → v2 load + save roundtrip preserves fields.
- Unit: nonce expiry + pruning.
- Integration: two-agent pair + accept over loopback. Verify
  peer_stores on both sides are mutated correctly with matching
  labels, origin, relationship.
- Integration: pair + subsequent `portl shell <paired-label>` works
  end-to-end without any manual trust configuration.
- Integration: invite revocation takes effect within reload window.
- Integration: pair with relay-hint populates peer entry + is
  surfaced in `portl peer ls`.
- Integration: pair over relay (both sides behind symmetric NAT,
  paired via v0.3.3 relay).

## 6. Migration / back-compat

### 6.1. Peer store schema

v1 → v2 is backward-compatible:

- v2 agent reads v1 files: treats missing `relay_hint` as `None`,
  missing `schema_version` as `1`.
- v1 agent reading v2 files: serde drops unknown fields, loads
  entries with no relay_hint. Functional; no security regression.
- First write by a v2 agent upgrades the file schema to v2.

### 6.2. CLI surface

No breaking changes:

- `peer add-unsafe-raw` remains. Pair protocol does NOT deprecate
  it; manual pinning is still the right answer for some cases
  (bootstrapping from existing eid knowledge, CI fixtures,
  recovery from network-partitioned pair failures).
- `peer invite`, `peer pair`, `peer accept` are additive.

### 6.3. Agent runtime

- Agent gains `portl/pair/v1` ALPN listener alongside existing
  `portl/ticket/v1`.
- `pending_invites.json` is new state; absent on upgrade, created
  lazily when `peer invite` is first run.

## 7. Security considerations

### 7.1. Invite code threat model

Invite codes are copy-pasteable strings distributed out-of-band.
Protections:

- **Nonce uniqueness**: single-use. Replayed code → `NonceUnknown`.
- **TTL**: default 1h, configurable. Lost codes don't linger forever.
- **Inviter's endpoint_id is bound in the code**: code works only
  against the specific inviter. Cannot pair with a different agent.
- **Caller identity verified by TLS**: even with a valid code, the
  acceptor records whoever's endpoint_id is in the TLS cert, not
  what the caller claims. No identity spoofing via the code.

### 7.2. Atomic vs. partial failures

If caller successfully mutates its local store but never receives
the response (network drop), the acceptor's store is the source of
truth. Caller's next `peer ls --active` will show connection works
but origin=raw (from the TLS peer cert fallback). Acceptor sees
origin=paired. Slight asymmetry but not a security bug.

### 7.3. Relay hints as trust vectors?

A paired peer could send a malicious `relay_hint` URL. Defenses:

- Relay URLs are hints, not policy. An operator's configured relays
  still take precedence when listed explicitly.
- All relay traffic is iroh-end-to-end-encrypted; the relay sees
  metadata but not content.
- v0.4 will add "relay allowlist per peer" for operators who want
  to reject hint-proposed relays.

## 8. Open questions

1. **Invite code character set**: strict base32 or base32hex for
   readability? Leaning strict (no 0/O or 1/I confusion) — matches
   iroh's existing ticket encoding.
2. **QR-code rendering**: `portl peer invite --qr` uses the same
   `qrcode` crate already vendored for ticket QR. Included in
   v0.3.4.
3. **Invite chaining / transitive trust**: explicit NO. Someone
   who wants A-via-B indirection can be paired with B, and B can
   pair A explicitly. No transitive semantics.
4. **Simultaneous invites (race condition)**: two pair attempts
   against the same nonce from different callers. First one wins
   (nonce consumed); second gets `NonceUnknown`. Acceptable.
5. **`peer accept` vs `peer pair --one-way`**: final call. Three
   verbs vs. two-verbs-with-flag. Going with three for clarity;
   reversible if user feedback disagrees.

## 10. Walkthrough: homelab self-host after v0.3.4

```bash
# ---- max (the server) ----
curl -fsSL https://raw.githubusercontent.com/KnickKnackLabs/portl/main/install.sh \
  | bash -s -- --agent --yes
# writes plist, starts agent, creates identity, seeds self-row

# Enable relay so peers behind symmetric NAT can still reach us
portl install --apply --yes \
    --with-relay \
    --relay-hostname relay.mynet.com \
    --relay-cert /etc/portl/relay.crt \
    --relay-key  /etc/portl/relay.key

# Issue invite for friend
portl peer invite --ttl 1h --for friend-laptop
# → code: PORTLINV-…

# ---- friend-laptop ----
# Share the code (DM, email); friend runs:
portl peer pair PORTLINV-…
# → adds max as mutual peer, learns max's relay hint

# Test: shell into max from friend-laptop
portl shell max
# → works immediately; no env vars, no ticket files, no copy-paste of eids
```

Total user surface visible in the happy path: two commands
(`install.sh`, `peer invite` on max; one command `peer pair` on
friend's machine) plus sharing one code string. Compared to v0.3.1's
add-unsafe-raw-plus-copy-paste-eid workflow, roughly half the
error-prone steps.
