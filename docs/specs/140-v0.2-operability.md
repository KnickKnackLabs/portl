## 1. Summary

v0.1.0 shipped the M0â€“M7 roadmap and proved the model end-to-end.
v0.2.0 closes two gaps the release revealed, on one tag, under one
theme: **Operability** â€” making portl fit to install in the morning
and forget about by afternoon. Two tracks, one release:

- **Operator experience** (Parts A + B) â€” shape the CLI, environment
  variables, and filesystem layout so that a fresh host goes from
  zero to "shelled into a container" in two commands. Subtracts
  surface: the `portl agent` namespace, `agent.toml`, the SQLite
  alias store, three-Dockerfile-emitting commands, two-knobs-for-one
  concept pairs, reachability probes in `doctor`.
- **Unattended-runtime safety** (Part C) â€” harden the shell
  handler's lifecycle so that sessions end cleanly, revocations
  propagate to live connections, disk exhaustion is bounded, and
  panics are surfaced to the supervisor. Adds contracts the v0.1
  runtime never promised.

Headline: `portl init && portl docker run <image>`. Two commands
from zero to inside-the-container.

Pre-1.0, with no external integrations. This is the only window in
which Part A + Part B's breaking changes are cheap. Part C is
non-breaking and ships first, ahead of v0.2 â€” see
[Â§14 Release sequence](#14-release-sequence).

## 2. Motivation

Concrete pain points from v0.1, each cross-referenced to the part
that addresses it:

**Operator experience (Parts A + B):**

1. Three separate ways to emit a Dockerfile, each with a different
   name, each solving a slightly different problem. (Part A, Â§6)
2. Three-word command paths for the common operations â€” `portl
   docker container add`, `portl agent run --mode gateway` â€” when
   the two- or one-word forms are unambiguous. (Part A, Â§4)
3. Two knobs for one concept: `portl mint-root` vs "delegate by
   re-signing"; `portl revoke` vs `portl revocations publish`;
   `portl id` subcommands vs everyone just wanting `init`.
   (Part A, Â§4)
4. `agent.toml` parsing covers a broader surface than operators
   actually set (audit in Â§8.2). The `toml` dependency pulls in
   transitive code every host runs. (Part B, Â§8)
5. `rusqlite-bundled` for a three-column alias store is the largest
   single dependency in the binary, the slowest crate to compile,
   and the **working hypothesis** (unverified) behind the macOS
   release-mode host crash. (Part B, Â§9)
6. `portl doctor` makes outbound TCP/443 probes against an iroh
   relay, so it is neither offline-safe nor deterministic.
   (Part B, Â§11)
7. Gateway HTTP bridging is always linked into `portl-agent`, even
   though it is slicer-specific and touched by â‰¤ 5 % of operators.
   (Part B, Â§10)
8. No uniform `portl install` / `portl init`; operators cobble
   systemd units from the CLI doc by hand. (Part A, Â§7)

**Unattended-runtime safety (Part C):**

9. Disconnect sends a single `SIGHUP` to the shell's signal target
   (`crates/portl-agent/src/shell_handler.rs:71, 756`). There is no
   escalation, no process-group reap, no grace window. Background
   jobs and daemonised children survive client drop. (Part C, Â§13.2)
10. Revocation records land in `revocations.jsonl` but **do not**
    terminate live sessions backed by the revoked ticket. The ticket
    is verified only at handshake. (Part C, Â§13.4)
11. `pty_stdout_thread` is a `std::thread::spawn` blocking read that
    outlives session teardown and holds the PTY master fd open.
    (Part C, Â§13.3)
12. `pre_exec` hooks exist on the non-PTY exec path only. There are
    no rlimits on the PTY shell path: a fork-bomb inside `portl
    shell` consumes the agent's uid's process budget. (Part C, Â§13.1)
13. `shell_spawn` is audited at `audit.rs:41`, but **spawn
    rejections** (path probe fails, uid lookup fails, user-switch
    refused) never emit `shell_exit`, breaking the invariant "one
    start â†” one exit per accepted session." (Part C, Â§13.5)
14. `revocations.jsonl` grows unbounded. A compromised operator
    (or a noisy delegation forest) can fill the disk.
    (Part C, Â§13.8)
15. Agent `SIGTERM` cancels the shutdown token but **does not walk
    live shell sessions** (`crates/portl-agent/src/lib.rs:223`).
    Service restarts orphan child processes. (Part C, Â§13.9)
16. Release profile does not set `panic = "abort"`. A panic on any
    async task terminates the task but leaves the process in a
    half-dead state the supervisor cannot detect. (Part C, Â§13.6)
17. No visibility into tokio runtime health. A wedged `spawn_blocking`
    call (e.g. a filesystem hang) is invisible until operators notice
    the agent is unresponsive. (Part C, Â§13.7)

## 3. Goals and non-goals

### In scope

**Operator experience (Parts A + B):**

- Collapse the operator CLI. One-word verbs wherever unambiguous.
  Multi-word only for adapter-scoped commands.
- `portl-agent` as the daemon; no `portl agent` subcommand namespace.
- Replace `agent.toml` with a small fixed env-var set, audited
  against every field currently used in tests or in-tree code.
- Replace `rusqlite` with an `fd-lock`'d JSON file. Concurrency
  semantics preserved (exclusive writers, shared readers).
- One Dockerfile for running portl as a service (`install
  dockerfile`); one command for bolting portl onto an existing
  image (`docker bake`); orchestrate-by-default for the common case
  where neither applies.
- Lift gateway mode out of `portl-agent` into a separate
  `portl-gateway` multicall entry + crate.
- `portl doctor` strictly local. `portl status --relay` performs
  real reachability.
- Autodetect-based `portl install` + `portl init`.

**Unattended-runtime safety (Part C):**

- Sessions end deterministically: client drop, agent stop, ticket
  revoke, and session-initiated exit all converge on a clean
  process-group reap within a bounded time budget.
- Resource consumption is bounded at spawn time: rlimits on every
  shell/exec path (both platforms, best effort noted in Â§13.1).
- Durability: audit records survive `kill -9`; revocation publish
  fails closed on ENOSPC; `revocations.jsonl` has a size ceiling.
- Supervisor-visible crashes: `panic = "abort"` + restart contract
  documented per platform.
- Runtime observability: slow blocking-task warnings land in
  journald + a metrics counter.

### Out of scope

- New wire protocols, new ALPNs, new capability fields.
- Changes to ticket schema v1. Chain semantics, signature scheme,
  canonical form, all preserved.
- Any backwards-compatibility shim for v0.1 config files or state
  layouts beyond the one-shot `aliases.sqlite` â†’ `aliases.json`
  conversion (see Â§14.2).
- `fs/v1`, VPN mode, multi-transport abstraction. Post-v0.2
  per `future/140-transport-abstraction.md`.
- Seccomp, landlock, PAM, chroot hardening. Redundant inside
  container/VM; deferred pending a concrete bare-host demand.
- PTY `--user` switch. Deferred to v0.3.

## 5. `portl docker` orchestrate

(Supersedes `060-docker.md Â§13â€“Â§15`.)

v0.1 assumed you build an image with portl as a layer, then ask the
adapter to run it. That is the **bake** case and v0.2 keeps it (Â§6).
Starting v0.2, the **orchestrate** case is the default, because it
is what the overwhelming majority of operators actually want: use
the existing image unchanged, inject a portl-agent process at
runtime, print a ticket.

### 5.1 Orchestrate-mode injection

`portl docker run <image>` does:

1. Generate an ephemeral ed25519 keypair on the host.
2. Mint a holder-bound ticket for the (not-yet-existing) endpoint id.
3. `docker create` the user's image with ENTRYPOINT/CMD **unchanged**.
   Args, env, volumes, network, user spec are passed through.
4. `docker start`.
5. `docker cp` the portl binary into the container (prefer
   `/usr/local/bin/portl`; fall back to `/tmp/portl` on read-only
   root, then `/dev/shm/portl` as a last resort).
6. `docker exec -d` the agent with the pre-generated identity and
   trust roots via env vars (see Â§8). No config file, no identity
   file inside the container.
7. Wait briefly for the agent to log `listening`. Print the ticket.

`portl docker attach <container>` skips steps 3â€“4 and injects into an
already-running container. `portl docker detach <container>` sends
`SIGTERM` to the bookkept exec id and removes any portl binary it
copied (leaves binaries that pre-existed the call alone).

### 5.2 Injection correctness

`docker exec` enters the target container's mount, pid, network,
user, cgroup, seccomp, AppArmor and capability namespaces. The
injected agent sees exactly what the target's main process sees, so
`portl shell`, `portl exec`, `portl tcp -L ...`, and `portl udp` all
behave the way operators expect. This is strictly more correct than
a sidecar container with `network_mode: service:X`, which shares
only the network namespace.

### 5.3 Architecture mismatch

Before step 5, the orchestrator reads `docker inspect .Architecture`.
If the target is a foreign architecture (qemu-user on Apple Silicon
is the common case), the embedded binary set is searched for a
matching `portl-<target>` blob (see Â§6.3). If none matches, the
orchestrator refuses with a clear error and suggests
`portl docker bake`.

### 5.4 Edge cases

| Situation | Behaviour |
| --- | --- |
| Read-only root FS | probe `/usr/local/bin/portl` â†’ `/tmp/portl` â†’ `/dev/shm/portl`; error with bake suggestion if all three refuse |
| Scratch / distroless-static | works (musl-static binary). `portl shell` fails for lack of `/bin/sh`; `portl exec /usr/bin/...` still works |
| Container restart | injected agent dies with the container. `portl docker run --watch` re-injects on Docker `die` events; `attach` is the one-shot form |
| Target runs as non-root | injected agent runs as that user. `portl shell` inherits the uid â€” correct |
| Target binary differs in arch | see Â§5.3 |

## 6. `portl docker bake`

Exactly two Dockerfile outputs, serving two distinct use cases:

| I want to... | Command |
| --- | --- |
| Run portl as a service in its own container (k8s, Nomad, etc) | `portl install dockerfile [--output DIR]` |
| Bolt portl onto my existing application image | `portl docker bake <base-image> [--tag T]` |

`portl install dockerfile` emits a Dockerfile whose ENTRYPOINT is
`portl-agent`. Base is `debian:stable-slim`, overridable with
`--base`. For running portl as its own service.

`portl docker bake <base-image>` emits a Dockerfile that extends the
user's image. ENTRYPOINT and CMD are **inherited from the base** by
default; only the binary is added. Users who want to run the agent
as an init-shim ahead of their original entrypoint pass
`--init-shim`, which generates a two-line ENTRYPOINT that execs
`portl-agent` as a subprocess and the original entrypoint as PID 1
under it. Opt-in only.

### 6.1 Output modes

- `portl docker bake <base> --output ./dir` writes a Dockerfile + a
  build context into `./dir`. No docker invoked.
- `portl docker bake <base> --tag myimg:portl` runs `docker build`
  via the local daemon and returns when the image is tagged.
- `portl docker bake <base> --tag myimg:portl --push` additionally
  pushes to the registry inferred from the tag.

### 6.2 Gone from v0.1

`portl docker image generate` and `portl docker image build` are
removed. Either emit a standalone portl-service image (use
`portl install dockerfile`) or bolt portl onto an existing image
(use `portl docker bake`). No third case.

### 6.3 Binary resolution

All `docker` / `install` commands that need a portl binary resolve
it in this order:

1. `--from-binary PATH` explicit override.
2. `std::env::current_exe()` â€” the binary running the CLI right
   now. Default.
3. `--from-release v0.2.0` â€” downloads
   `portl-v0.2.0-<target>.tar.zst` from the GitHub release,
   verifies the sha256 sidecar, uses the extracted binary.

For foreign-arch injection (Â§5.3), portl invokes option 3 at runtime
so a macOS-host orchestrator can inject a linux-x86_64 agent without
the operator staging the tarball manually. Requires network access;
falls back with `--from-binary` suggestion on failure.

## 7. `portl init` and `portl install`

### 7.1 `portl init`

```
portl init [--force] [--role operator|agent]
```

Composes:

1. If no identity, generate + persist.
2. Run `portl doctor`; pretty-print the results.
3. If `--role agent` or autodetected as a long-running host (systemd
   present, non-laptop hostname, `$PORTL_HOME` writable under
   `/var/`), suggest `portl install`.
4. Print the first-mint cookbook:
   `portl docker run <image>` / `portl slicer run <image>`.

### 7.2 `portl install`

```
portl install [TARGET] [--apply] [--yes] [--detect] [--dry-run]
```

Targets: `systemd`, `launchd`, `dockerfile`, `openrc`. When `TARGET`
is omitted, the command autodetects (Â§7.3) and either prompts to
confirm or, with `--yes`, proceeds. `--detect` prints a JSON
description of what was found and exits 0.

With `--apply`, the command:

- Copies `current_exe()` to the target install path.
- Writes the unit / plist / script to the system location.
- Validates with `systemd-analyze verify` / `plutil -lint` / etc.
- Reloads the init system and starts the service.
- Tails the service log for 5 s to confirm startup.

Generated systemd units include `EnvironmentFile=-/etc/portl/agent.env`
(optional) so operators who want file-based config discipline have
a first-class path (Â§8).

### 7.3 Autodetect heuristics

In order of strength, first match wins:

| Signal | Target |
| --- | --- |
| `/bin/launchctl` exists + `uname -s` is `Darwin` | `launchd` |
| `/.dockerenv` exists or `/proc/1/cgroup` contains container markers | refuse; suggest `portl docker bake` or explicit `--target` |
| `/run/systemd/system/` is a directory | `systemd` |
| `/sbin/openrc-run` exists | `openrc` (warn: less tested) |
| otherwise | fail with an explicit list of supported targets |

Running as non-root on a systemd host picks user-level systemd
(`~/.config/systemd/user/portl-agent.service`) rather than refusing.

### 9.1 Alias store semantics

v0.1 used `aliases.sqlite` with SQLite WAL + busy-timeout for
concurrent-writer safety (`crates/portl-cli/src/alias_store.rs:11`,
concurrent-write test at line 385). v0.2 replaces it with a JSON
file protected by `fd-lock` (advisory POSIX file locking):

- Readers take a **shared** lock for the duration of their read.
- Writers take an **exclusive** lock for the read-modify-write cycle.
- On `ENOSPC` during rename, the old file is left intact and the
  write fails closed; callers surface the error.

This is a real behaviour change: concurrent writers now serialize
at a coarser granularity than SQLite's WAL (whole-file lock instead
of per-row). For a three-column key-value store with â‰¤ 1 kHz write
rate (alias additions on human timescales), this is irrelevant.
The contract operators see is the same: concurrent CLIs do not
corrupt each other's writes, lost-write races that were possible
under v0.1 are still possible under v0.2. Documented explicitly in
`CHANGELOG.md` on v0.2 ship.

## 10. Lift gateway out of `portl-agent`

The slicer HTTP gateway (`portl agent run --mode gateway
--upstream-url ...`) becomes a top-level command: `portl gateway
<upstream-url>`. Code moves from `crates/portl-agent/src/gateway/*`
to a new crate `crates/portl-gateway/` which depends on
`portl-proto` + `portl-core` but **not** on `portl-agent`.

The multicall binary gains a third entrypoint: `portl-gateway`,
symlinked to the same ELF. Systemd units that want gateway mode
invoke `ExecStart=/usr/local/bin/portl-gateway
https://slicer.example.com`. Agent units invoke
`ExecStart=/usr/local/bin/portl-agent`. They are deployed
independently.

Consequences:

- `portl-agent` loses `reqwest` as a direct dep (gateway was the
  only user). â‰ˆ 400 kB of binary shaved.
- Ticket schema stops carrying bearer-specific logic in the core
  library. Master ticket bearer validation, VCHAR check, widening
  refusal, and `to.is_some()` enforcement all move into
  `portl-gateway::master_ticket`. The ticket crate gets smaller
  and more purely about authz.

## 13. Session lifecycle and daemon invariants

The invariants below are what a v0.2 agent promises to the host and
the operator, regardless of what the shell session did. They are
**contracts**, not implementation steps â€” the plan in
`../../CHANGELOG.md` records how each was decomposed into
bite-sized tasks. Three of the nine (Â§13.1, Â§13.5, Â§13.6) ship
early in v0.1.1 (see Â§14).

### 13.1 Resource limits on every spawn path

Every `portl shell` and `portl exec` invocation spawns its child
under the following `setrlimit(2)` values, applied in the `pre_exec`
hook **before** `execve`:

| Resource | Limit | Rationale |
| --- | --- | --- |
| `RLIMIT_NOFILE` | `4096` | Prevents fd-exhaustion DoS; 4096 is comfortably above any interactive workload. |
| `RLIMIT_NPROC` | `512` | Bounds per-uid process count. Fork-bomb containment on Linux. |
| `RLIMIT_CORE` | `0` | No core dumps from guest processes (avoids leaking memory to disk). |
| `RLIMIT_CPU` | `86_400` (24h) | Runaway compute bound. Tunable via env in v0.3. |
| `RLIMIT_FSIZE` | `10 GiB` | Caps a single file write; avoids filling the host disk accidentally. |

**Platform caveat (Linux vs macOS).** `RLIMIT_NPROC` is per-uid on
Linux (what we want â€” a fork-bomb hits the process-count cap for
the agent's effective uid). On Darwin it is per-process only, which
means the fork-bomb acceptance gate shipped in `v0.1.1`
is **Linux-specific**. macOS gets `RLIMIT_NOFILE` / `RLIMIT_CORE` /
`RLIMIT_CPU` / `RLIMIT_FSIZE` but not the fork-bomb cap. This is
documented in CHANGELOG as a known platform limitation.

**Implementation note.** v0.1 has a `pre_exec` hook only on the
non-PTY exec path (`crates/portl-agent/src/shell_handler.rs:327,
785-804`). The PTY shell path uses `portable_pty::CommandBuilder`,
which has no equivalent. v0.2 replaces the PTY spawn with a direct
`CommandExt::pre_exec` implementation that retains PTY allocation
via `openpty(3)` + `setsid` + `ioctl(TIOCSCTTY)` in the hook.

### 13.2 Process-group kill escalator on session drop

When a shell or exec session ends (client disconnect, stream error,
or explicit close), the agent walks the child's **process group**
with the sshd-style escalator:

| Step | Signal | Grace | Purpose |
| --- | --- | --- | --- |
| 1 | `SIGHUP` to pgroup | 5 s | Interactive shells terminate their jobs |
| 2 | `SIGTERM` to pgroup | 5 s | Non-interactive children cooperate |
| 3 | `SIGKILL` to pgroup | â€” | Uncooperative survivors |

The escalator runs as a bounded-time async task spawned from
`ShellSessionGuard::drop` (`crates/portl-agent/src/shell_handler.rs:71`).
A session can report "cleanly exited" at any step and skip the
remainder.

**Invariant.** No session leaves process-group descendants alive
past `SIGKILL + 100 ms`. Background jobs that have been explicitly
daemonised (double-forked out of the session's pgroup) are out of
scope â€” they will survive, matching sshd behaviour.

### 13.3 PTY output drain with force-close

v0.1 spawns `pty_stdout_thread` as a `std::thread::spawn` blocking
read from the PTY master fd (`crates/portl-agent/src/shell_handler.rs:481`).
On session teardown that thread can outlive the session and hold
the PTY master fd open indefinitely.

v0.2 moves the PTY reader onto a tokio task backed by `AsyncFd`.
On session close:

- Normal exit: the reader drains up to 30 s of remaining PTY output,
  then closes.
- Forced close: after 30 s or on explicit `force_close` from the
  session manager, the reader's `AsyncFd` is dropped and the master
  fd is closed. Any bytes still in kernel buffers are lost â€” this
  is by design, matching sshd's behaviour on `ClientAliveInterval`
  timeout.

**Invariant.** At most one PTY master fd per shell session;
session end releases the fd within 30 s.

### 13.4 Revocation kills live sessions

v0.1 checks ticket validity only at handshake. Revocations land in
`revocations.jsonl` but do not terminate running sessions that
authenticated under a now-revoked ticket.

v0.2 wires a per-session `CancellationToken` into the shell
registry. `RevocationSet::insert` fires every matching session's
token, which triggers the Â§13.2 escalator (`SIGHUP` â†’ `SIGTERM` â†’
`SIGKILL`). The revoker sees `--publish` complete only **after**
the local registry has scheduled the cancellation (not after the
cancellation completes â€” that could take up to 10 s).

**Invariant.** A revocation committed to disk is either (a) signalled
to every live session backed by the revoked ticket within 2 s, or
(b) rolled back on `ENOSPC` before ack (Â§13.8).

### 13.5 Balanced audit: start and exit

v0.1 emits `audit.shell_spawn` at `audit.rs:41` for every accepted
session, and `audit.shell_exit` at `audit.rs:52`. The asymmetry:
**spawn rejections** (path probe fails, uid lookup fails, user
switch refused) log at `warn!` level but never emit `shell_exit`.

v0.2 invariant: **every accepted session emits exactly one
`shell_start` followed by exactly one `shell_exit`.** Rejections
before spawn emit a separate `shell_reject` record with the
rejection reason; they do not emit a spawn/exit pair.

The `shell_start` record includes: `ticket_id`, `caller_endpoint_id`,
`session_id`, `pid`, `mode` (pty|exec), `shell_user`, `shell_argv0`.
The `shell_exit` record adds `exit_code` and wall-time duration.

### 13.6 `panic = "abort"` and the supervisor contract

Release profile sets `panic = "abort"`. A panic anywhere in the
agent's async runtime terminates the process non-zero. The
supervisor (systemd / launchd / Docker restart policy) restarts it
under its declared `Restart=on-failure` + `RestartSec` policy.

**Pre-work required before enabling this flag.** The agent has a
handful of latent panic sites (scattered `.expect(...)`, mutex
poison â†’ `anyhow!` that only returns; not all are caller-tolerated).
v0.2 includes a panic-site audit (`grep -rn 'expect\|unwrap' src/`
+ review) before flipping `panic = "abort"` on. See
`../../CHANGELOG.md` for the shipped task breakdown.

**Supervisor-side requirement.** Units emitted by
`portl install systemd|launchd` set `Restart=on-failure`,
`RestartSec=5s`, `StartLimitBurst=5`, `StartLimitIntervalSec=60s`.
On OpenRC the equivalent `supervise-daemon --respawn-delay 5
--respawn-max 5` flags are set. Docker images use
`--restart=unless-stopped`.

### 13.7 Slow-task detection

Tokio's default runtime has no visibility into `spawn_blocking`
tasks that hang. v0.2 wraps every `spawn_blocking` call site in a
`portl_core::runtime::slow_task` helper that:

- Starts a timer before calling into the closure.
- If the closure runs > 60 s, emits a `warn!` log with the call-site
  label and the elapsed time so far.
- On return, increments `slow_tasks_total{label=...}` on the metrics
  socket.

Call sites to be wrapped include the PTY child-wait and
`RevocationSet::append`. A pre-commit lint (simple grep) catches
any new `spawn_blocking` call sites that bypass the wrapper.

### 13.8 `revocations.jsonl` size ceiling

v0.1 appends indefinitely. A compromised operator or noisy
delegation forest fills the disk. v0.2 imposes:

- Size ceiling `PORTL_REVOCATIONS_MAX_BYTES` (default `10 MiB`).
- On overflow attempt: the append fails with a dedicated error
  variant and `meta/v1 PublishRevocations` replies `ResourceExhausted`
  to the caller. The daemon does **not** auto-rotate (Â§13 principle:
  fail closed rather than silently forget revocations).
- Rotation is an operator concern: `portl revoke --compact` folds
  revocations for tickets that are also past their expiry into a
  single "expired-and-revoked" bucket, reducing file size
  significantly in typical steady-state forests.

GC policy is unchanged: a revocation record is removable once
`revoked_at + LINGER < now` AND the ticket itself has expired
(070-security.md Â§4.12).

### 13.9 Graceful agent shutdown

v0.1's `install_signal_tasks` (`crates/portl-agent/src/lib.rs:223`)
cancels a shutdown token on `SIGTERM` / `SIGINT` but does not walk
live sessions.

v0.2 invariant on `SIGTERM`:

1. Endpoint stops accepting new connections immediately.
2. Each live shell session receives the Â§13.2 escalator in parallel.
3. The audit log `shell_exit` record is flushed (fsync) for every
   session before exit.
4. Process exits 0 if all sessions reap within 30 s; exits non-zero
   (so the supervisor notices) and force-kills survivors otherwise.

SIGKILL of the agent is of course out of scope â€” no user-space
teardown is possible.
