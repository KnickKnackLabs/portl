# v0.2.0 Operability Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use
> superpowers:subagent-driven-development (recommended) or
> superpowers:executing-plans to implement this plan task-by-task.
> Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship v0.2.0 Operability. Parts A + B (minus §9, shipped
in v0.1.2) + the remaining Part C items. Spec:
[`../specs/140-v0.2-operability.md`](../specs/140-v0.2-operability.md).

**Release theme:** Unattended-operability. Breaking at the CLI,
env-var, and filesystem-layout surfaces. No user data migration
tooling — pre-1.0, maintainer-only. v0.1 identity files are
untouched (ticket schema, ALPNs, handshake unchanged).

**Execution strategy:** Three sequential phases, each executed by
a single worker subagent. Each phase ends with a local green CI
run before handing off to the next phase. Final roundtable review
after Phase 3 lands.

| Phase | Theme | Sections from spec 140 |
| --- | --- | --- |
| 1 | Internals cleanup | §8 env-var schema · §10 gateway lift · §11 local doctor |
| 2 | Runtime safety | §13.2 pgroup escalator · §13.3 PTY async drain · §13.4 revoke kills live · §13.7 slow-task detection · §13.8 revocations size ceiling · §13.9 graceful shutdown |
| 3 | Operator experience | §4 CLI collapse · §5 docker orchestrate · §6 docker bake · §7 init / install · §12 multicall-only agent + shim |

Each phase may be one or many commits at the worker's discretion,
but each commit must pass `cargo fmt --check`, `cargo clippy
--workspace --all-features --all-targets -- -D warnings`, and
`cargo nextest run --profile ci --workspace --all-features`.

### Task 1.2 — `portl-gateway` multicall entrypoint (§10, scoped)

> **Scope note**: the spec's full §10 crate lift is larger than
> it looked on paper. Gateway code in v0.1 is entangled with
> `portl-agent` internals (`AgentState`, `Session`,
> `caps_enforce`, `stream_io`, `audit`) and master-ticket logic
> lives in `portl-core`/`portl-proto`. Fully extracting these
> is a ≥19-file refactor that risks destabilising the tests the
> rest of v0.2.0 depends on. **For v0.2.0 we ship the
> user-observable slice of §10 only**: the `portl-gateway`
> multicall binary + the CLI-level contract change. The
> crate-boundary lift (moving gateway code out of `portl-agent`,
> dropping `reqwest` from `portl-agent`'s direct deps, stopping
> the ticket crate from knowing about master tickets) is
> deferred to v0.2.1 and captured in `autoresearch.ideas.md`.
>
> The operator sees the same CLI surface — `portl-gateway
> <upstream>` is a valid way to run the daemon in gateway mode —
> and the binary-size / build-time wins are deferred by one
> release.
>
> **Rationale**: v0.2.0 already ships breaking changes across
> CLI, env-vars, docker adapter, runtime teardown, and session
> cancellation. Adding a full crate restructure to the same tag
> multiplies the probability of a regression landing that
> requires a v0.2.1 release anyway. Shipping §10 as a two-step
> sequence — user-observable behaviour in v0.2.0, internal
> rearrangement in v0.2.1 — is cheaper and keeps the reversible
> footprint small.

**Files (v0.2.0 scope):**
- Modify: `crates/portl-cli/src/lib.rs::rewrite_multicall` — add
  a third case. When `argv[0]` basename is `portl-gateway`,
  rewrite to `["portl", "gateway", ...rest]` (matching the
  existing `portl-agent` handler style).
- Modify: `crates/portl-cli/src/lib.rs` Cli tree — ensure a
  top-level `portl gateway <upstream-url>` command exists that
  drives the gateway mode. Under the hood it may still call the
  gateway code that lives in `portl-agent`; that's fine for
  v0.2.0.
- Modify: `.github/workflows/ci.yml` `release-build` job — the
  packaging step creates a `portl-gateway` symlink inside each
  tarball alongside `portl-agent` (same ELF). Adjust the
  tar creation + shasum steps to include the new symlink.
- Document in `autoresearch.ideas.md` at repo root: "v0.2.1 —
  lift gateway into portl-gateway crate; remove reqwest/wiremock

**Tests required:**
- `portl_gateway_binary_accepts_upstream_arg` — integration test

**Acceptance gates (v0.2.0 scope):**

## Phase 2 — Runtime safety

**Spec sections:** 140 §13.2, §13.3, §13.4, §13.7, §13.8, §13.9.
**Goal:** Ship the Part C invariants that couldn't slip into
v0.1.1's safety-net release.

### Task 2.1 — Process-group kill escalator on session drop (§13.2)

When a shell / exec session ends (client drop, stream error,
explicit close), the agent walks the child's **process group**
via the sshd-style escalator:

| Step | Signal | Grace | Purpose |
| --- | --- | --- | --- |
| 1 | `SIGHUP` to pgroup | 5 s | Interactive shells hang up |
| 2 | `SIGTERM` to pgroup | 5 s | Non-interactive children cooperate |
| 3 | `SIGKILL` to pgroup | — | Uncooperative survivors |

**Files:**
- Modify: `crates/portl-agent/src/shell_handler.rs` — extract a
  new `SessionReaper` struct that owns the child's pgid and
  implements the escalator. Called from `ShellSessionGuard::drop`
  (or its async teardown path).
- Invariant: `killpg(pgid, SIGHUP)` → wait ≤ 5 s → `killpg(pgid,
  SIGTERM)` → wait ≤ 5 s → `killpg(pgid, SIGKILL)`. Skip remaining
  steps if the session is observed reaped.

**Tests required:**
- `session_reaper_kills_interactive_shell_on_hup` (Linux).
- `session_reaper_escalates_to_term_on_hup_ignored`.
- `session_reaper_escalates_to_kill_on_term_ignored`.
- `background_jobs_in_pgroup_are_terminated`.
- `double_forked_daemons_survive_teardown` (matches sshd).

### Task 2.2 — PTY output drain with force-close (§13.3)

**Files:**
- Modify: `crates/portl-agent/src/shell_handler.rs` — replace the
  `std::thread::spawn` that reads the PTY master fd with a tokio
  task backed by `tokio::io::unix::AsyncFd`.
- New: on session close, drain up to 30 s of remaining PTY
  output; on `force_close` or timeout, drop the `AsyncFd` and
  close the master fd. Any buffered kernel bytes are lost
  (matches sshd).

**Invariant:** at most one PTY master fd per shell session;
session end releases the fd within 30 s.

**Tests required:**
- `pty_drain_completes_on_normal_exit`.
- `pty_drain_force_closes_at_30s`.
- `pty_master_fd_closed_after_session_end`.

### Task 2.3 — Revocation kills live sessions (§13.4)

**Files:**
- Modify: `crates/portl-agent/src/revocations.rs` — `RevocationSet`
  gains a registry of live-session `CancellationToken`s keyed by
  the ticket-id (full or chain-prefix) they were opened under.
- Modify: `crates/portl-agent/src/session.rs` (or equivalent) —
  every accepted session registers its cancellation token with
  the `RevocationSet` on accept; deregisters on close.
- On `RevocationSet::insert(record)`: iterate matching tokens,
  cancel each. The cancellation triggers the §13.2 escalator via
  the session's teardown path.

**Invariant:** a revocation committed to disk is either (a)
signalled to every live matching session within 2 s, or (b) the
disk commit failed (ENOSPC → `PublishRevocations` replies with
`ResourceExhausted`; §13.8).

**Tests required:**
- `revocation_cancels_live_shell_session_within_2s`.
- `revocation_of_parent_ticket_cancels_delegated_session`.
- `enospc_revocation_returns_resource_exhausted`.

### Task 2.4 — Slow-task detection (§13.7)

**Files:**
- Create: `crates/portl-core/src/runtime.rs` — new helper
  `slow_task(label: &'static str, fut) -> impl Future` that
  wraps any `spawn_blocking` closure:
  - Starts a timer before the closure runs.
  - If the closure runs > 60 s, emits `warn!` with the label and
    elapsed time.
  - On return, increments a metrics counter
    `slow_tasks_total{label=...}`.
- Modify: every `spawn_blocking` site in the agent to wrap via
  `slow_task`. Call sites (to audit): PTY child-wait,
  `RevocationSet::append`, any `std::fs::*` calls under
  `spawn_blocking`.
- Add: a pre-commit lint or a `grep` CI check that fails if a
  new `spawn_blocking` call site does not use the wrapper.

**Tests required:**
- `slow_task_warns_after_60s`.
- `slow_task_increments_counter_on_return`.
- CI grep: `grep -rn 'spawn_blocking' crates/portl-agent | grep -v slow_task` returns empty.

### Task 2.5 — `revocations.jsonl` size ceiling (§13.8)

**Files:**
- Modify: `crates/portl-agent/src/revocations.rs` — add
  `PORTL_REVOCATIONS_MAX_BYTES` env var (default `10 MiB`).
- `RevocationSet::append`: before the actual append, check file
  size; if the append would exceed the ceiling, fail with a
  dedicated error variant `AppendError::SizeCeilingExceeded`.
- Modify: `meta/v1` handler — `PublishRevocations` replies with
  `ResourceExhausted` when the backing append fails closed.
- New CLI: `portl revoke --compact` folds revocations for tickets
  that are past their expiry into a single "expired-and-revoked"
  bucket (keeps the size bounded in steady-state). Can be
  deferred to v0.2.1 if too much scope; note in plan.

**Tests required:**
- `append_at_ceiling_returns_size_exceeded_error`.
- `publish_revocations_replies_resource_exhausted_on_ceiling`.
- `portl_revoke_compact_removes_expired_and_revoked_entries`
  (only if `--compact` ships in v0.2.0; otherwise add a stub
  that documents the v0.2.1 deferral).

### Task 2.6 — Graceful agent shutdown (§13.9)

**Files:**
- Modify: `crates/portl-agent/src/lib.rs::install_signal_tasks`
  — on `SIGTERM` / `SIGINT`:
  1. Endpoint stops accepting new connections (cancel accept loop).
  2. Walk live shell-session registry; invoke the §13.2
     escalator on each in parallel.
  3. Flush (`fsync`) the audit log `shell_exit` record for every
     session before exiting.
  4. Process exits 0 if all sessions reap within 30 s;
     non-zero otherwise (supervisor notices + force-kills).

**Tests required:**
- `sigterm_reaps_live_sessions_and_exits_zero_within_30s`.
- `sigterm_with_stuck_session_force_kills_and_exits_nonzero`.
- `audit_shell_exit_is_fsynced_before_agent_exits`.
