# 140 — v0.2.0 Cleanup: Breaking Simplifications

> Status: **proposed**, targeting v0.2.0. Supersedes parts of `080-cli.md`,
> `090-config.md`, and the post-v0.1 section of `120-roadmap.md`.
> No backward-compatibility shims are planned; see
> [`§13 Migration from v0.1`](#13-migration-from-v01).

## 1. Summary

v0.1.0 shipped a full roadmap (M0–M7) and proved the model end to end,
but the CLI surface, environment-variable layout, and dependency set
accumulated mud along the way. Now is the only window in which breaking
shape changes are cheap. v0.2.0 is a **cleanup release**: no new
protocols, no new ALPNs, no new capabilities. Pure subtraction.

Headline user experience: a fresh host installs portl, runs
`portl init`, then `portl docker run <image>`, and is shelled in. Two
commands from zero to "inside the container."

## 2. Motivation

Concrete pain points from v0.1, each of which motivates a specific
cleanup below:

1. Three separate ways to emit a Dockerfile, all of them named
   differently, each solving a slightly different problem.
   (`§9`)
2. Three-word command paths for the most common operations —
   `portl docker container add`, `portl agent run --mode gateway` —
   when the two- or one-word equivalents are unambiguous.
   (`§4`, `§8`, `§10`)
3. Two knobs for one concept: `portl mint-root` vs "delegate by
   re-signing"; `portl revoke` vs `portl revocations publish`;
   `portl id` subcommands vs everyone just wanting `init`.
   (`§4`)
4. `agent.toml` parsing code path that only ever sets four fields in
   anger, plus the `basic-toml` dependency it pulls in. (`§5`)
5. `rusqlite` for a ~3-column key-value alias store — the largest
   single dependency in the binary, the slowest crate to compile, and
   the working hypothesis behind the macOS release-mode crash. (`§6`)
6. `portl doctor` makes outbound network calls and so is neither
   offline-safe nor deterministic. (`§11`)
7. Gateway HTTP bridging (master tickets, bearer validation, per-
   connection header injection) is always linked into `portl-agent`,
   even though it is a slicer-specific feature that ≤ 5 % of operators
   will ever touch. (`§10`)
8. `portl agent run --ephemeral` — a mode flag we never shipped but
   obviously need for the container-orchestrate path (`140 §8.3`
   below). Better: make it implicit from the env. (`§5`)

## 3. Goals and non-goals

### In scope

- Collapse the operator CLI to one flat level where possible.
- Remove `portl agent` as a CLI namespace; daemon is `portl-agent` only.
- Replace `agent.toml` with a small fixed env-var set.
- Replace `rusqlite` with a JSON file for the alias store.
- Unify the three Dockerfile-emitting commands into two.
- Lift gateway mode out of `portl-agent` into its own multicall entry.
- Make `portl doctor` strictly local; promote reachability checks to
  `portl status --relay`.
- Orchestrate-by-default for `portl docker` and `portl slicer`; bake
  is opt-in. (See `140-v0.2-cleanup.md §8` for the model and
  `060-docker.md §14` which this doc supersedes.)
- Autodetect-based `portl init` and `portl install`.

### Out of scope

- New wire protocols, new ALPNs, new capability fields.
- Changes to ticket schema v1; chain semantics, signature scheme,
  encoding are all preserved.
- Any kind of compatibility shim for v0.1 CLIs, configs, or state
  layouts. A v0.1 operator will need a short migration step; see
  `§13`.
- `fs/v1`, VPN mode, multi-transport abstraction. Those remain
  post-v0.2 per `future/140-transport-abstraction.md`.

## 5. Environment and configuration

`agent.toml` is removed. The persistent daemon reads exactly these
environment variables, and nothing else:

| Env var | Default | Meaning |
| --- | --- | --- |
| `PORTL_HOME` | ProjectDirs data dir | State directory. If it contains an `identity.bin`, the agent runs in persistent mode. |
| `PORTL_IDENTITY_SECRET_HEX` | *unset* | If set, agent runs ephemeral: identity is the decoded 32 bytes, nothing is persisted. Mutually exclusive with `PORTL_HOME/identity.bin`. |
| `PORTL_TRUST_ROOTS` | *unset* | Comma-separated hex operator pubkeys. Required in ephemeral mode; optional in persistent mode where it augments any roots already stored. |
| `PORTL_LISTEN_PORT` | `0` | UDP port for the iroh endpoint. `0` = ephemeral. |
| `PORTL_DISCOVERY` | `dns,pkarr,local,relay` | Comma list of discovery backends. `none` disables discovery entirely. |
| `PORTL_METRICS` | `1` persistent, `0` ephemeral | Metrics unix socket at `$PORTL_HOME/metrics.sock`. Disabled automatically when `PORTL_HOME` is a tmpfs. |

Operators who want file-based config discipline use systemd
`EnvironmentFile=` or `docker run --env-file`. `portl install systemd`
(see `§12`) emits a unit that loads `/etc/portl/agent.env` if it
exists.

Removed env vars from v0.1: `PORTL_AGENT_CONFIG`, `PORTL_METRICS_SOCK`
(always at `$PORTL_HOME/metrics.sock`), and all of `PORTL_AGENT_*`
except where listed above.

The persistent-vs-ephemeral distinction is now **entirely implicit**.
No flag, no `--mode` switch. The agent picks the mode from the
environment and logs which one it chose at startup.

## 6. Filesystem layout

```
$PORTL_HOME/
├── identity.bin            # ed25519 secret, 0600
├── aliases.json            # operator-side: alias → endpoint_id/ticket map (replaces aliases.db)
├── revocations.jsonl       # unchanged
└── metrics.sock            # unix socket, 0600 (was PORTL_METRICS_SOCK, now fixed path)
```

Ephemeral mode creates `$PORTL_HOME` on tmpfs (`/run/portl` when
available, otherwise `tempfile::tempdir`), writes nothing but an
in-memory view of `identity`, and cleans up on `SIGTERM`.

`aliases.db` (SQLite) is gone. See `§7`.

## 8. Cleanup detail: `portl docker` orchestrate

(Supersedes `060-docker.md §13–§15`.)

v0.1 assumed you would build an image with portl as a layer, then
ask the adapter to run it. That's the **bake** case. Starting v0.2,
the **orchestrate** case is the default, because it is what the
overwhelming majority of operators actually want.

### 8.1 Orchestrate-mode injection

`portl docker run <image>` does:

1. Generate an ephemeral ed25519 keypair on the host.
2. Mint a holder-bound ticket for the (not-yet-existing) endpoint id.
3. `docker create` the user's image with the user's ENTRYPOINT/CMD
   **unchanged**. The user's args, env, volumes, network, user spec
   are all passed through.
4. `docker start`.
5. `docker cp` the portl binary into the container (`/usr/local/bin/portl`
   preferred, `/tmp/portl` on read-only root, `/dev/shm/portl` as
   last resort).
6. `docker exec -d` the agent with the pre-generated identity and
   trust roots supplied as env vars. No config file, no identity
   file inside the container.
7. Wait briefly for the agent to log `listening`. Print the ticket.

`portl docker attach <container>` skips steps 3–4 and injects into an
already-running container. `portl docker detach <container>` sends
the agent `SIGTERM` via a bookkept exec id and removes any portl
binary it added (leaves binaries that pre-existed the call alone).

### 8.2 The `--ephemeral` flag that never was

The agent auto-selects mode from `PORTL_IDENTITY_SECRET_HEX` presence
(see `§5`). The injected invocation always exports that var; the
host-installed invocation never does. No flag, no branches in the CLI.

### 8.3 Injection correctness

`docker exec` enters the target container's mount, pid, network,
user, cgroup, seccomp, AppArmor and capability namespaces. The
injected agent sees exactly what the target's main process sees, so
`portl shell`, `portl exec`, `portl tcp -L ...`, and `portl udp` all
behave the way operators expect. This is strictly more correct than
a sidecar container with `network_mode: service:X`, which shares only
the net namespace.

### 8.4 Architecture mismatch

Before step 5, the orchestrator reads `docker inspect .Architecture`.
If the target container is a foreign architecture (qemu-user on
Apple Silicon), the embedded binary set is searched for a matching
`portl-<target>` blob (see `§9.3` for how those are embedded); if
none matches, the orchestrator refuses with a clear error and
suggests `portl docker bake` for that image.

### 8.5 Edge cases

| Situation | Behaviour |
| --- | --- |
| Read-only root FS | probe `/usr/local/bin/portl` → `/tmp/portl` → `/dev/shm/portl`; error with bake suggestion if all three refuse |
| Scratch / distroless-static | works (musl-static binary). `portl shell` fails for lack of `/bin/sh`; `portl exec /usr/bin/...` still works |
| Container restart | injected agent dies with the container. `portl docker run --watch` re-injects on Docker `die` events; `attach` is the one-shot form |
| Target runs as non-root | injected agent runs as that user. `portl shell` inherits the uid — correct |
| Target binary differs in arch | see `§8.4` |

## 9. Cleanup detail: `portl docker bake`

(Supersedes the three Dockerfile-emitting commands in v0.1.)

### 9.1 Exactly two Dockerfile outputs

| I want to... | Command |
| --- | --- |
| Run portl as a service in its own container (k8s, Nomad, etc) | `portl install dockerfile [--output DIR]` |
| Bolt portl onto my existing application image | `portl docker bake <base-image> [--tag T]` |

`portl install dockerfile` emits a Dockerfile whose ENTRYPOINT is
`portl-agent`. Base is `debian:stable-slim`, overridable with
`--base`. This is for running portl as its own service.

`portl docker bake <base-image>` emits a Dockerfile that extends the
user's image. ENTRYPOINT and CMD are **inherited from the base** by
default; only the binary is added. Users who want to run the agent
as an init-shim ahead of their original entrypoint pass
`--init-shim`, which generates a two-line ENTRYPOINT that execs
`portl-agent` as a subprocess and the original entrypoint as PID 1
under it. Opt-in only.

### 9.2 `bake` has --output and --tag modes

- `portl docker bake <base> --output ./dir` writes a Dockerfile + a
  build context into `./dir`. No docker invoked.
- `portl docker bake <base> --tag myimg:portl` runs a `docker build`
  via the local daemon and returns when the image is tagged.
- `portl docker bake <base> --tag myimg:portl --push` additionally
  pushes to the registry inferred from the tag.

### 9.3 Where does the binary come from

All three `docker` / `install` commands that need a binary resolve it
in this order:

1. `--from-binary PATH` explicit override.
2. `std::env::current_exe()` — the binary running the CLI right now.
   Default.
3. `--from-release v0.2.0` — downloads
   `portl-v0.2.0-<target>.tar.zst` from the GitHub release, verifies
   the sha256 sidecar, uses the extracted binary.

For foreign-arch injection (`§8.4`), `portl` ships with the ability
to invoke option 3 at runtime, so a macOS-host orchestrator can
inject a linux-x86_64 agent without requiring the operator to stage
the tarball manually. Network access required; falls back to an
error suggesting `--from-binary`.

## 10. Cleanup detail: lift gateway out of `portl-agent`

The slicer HTTP gateway (`portl agent run --mode gateway
--upstream-url ...`) becomes a top-level command: `portl gateway
<upstream-url>`. Code moves from `crates/portl-agent/src/gateway/*`
to a new crate `crates/portl-gateway/` which depends on
`portl-proto` + `portl-core` but **not** on `portl-agent`.

The multicall binary gains a third entrypoint: `portl-gateway`,
symlinked to the same ELF. Systemd units that want gateway mode
invoke `ExecStart=/usr/local/bin/portl-gateway
https://slicer.example.com`. Agent units invoke
`ExecStart=/usr/local/bin/portl-agent`. They are deployed
independently.

Consequences:

- `portl-agent` loses `reqwest` as a direct dep (gateway was the only
  user). Roughly 400 kB of binary shaved.
- Ticket schema stops carrying bearer-specific logic in the core
  library. Master ticket bearer validation, VCHAR check, widening
  refusal, and `to.is_some()` enforcement all move into
  `portl-gateway::master_ticket`. The ticket crate gets smaller and
  more purely about authz.
- `wiremock` moves with the gateway tests into the new crate.

## 11. Cleanup detail: `portl doctor` is strictly local

v0.1's doctor makes a TCP/443 probe against
`use1-1.relay.iroh.network`. That's replaced:

| Check | v0.1 location | v0.2 location |
| --- | --- | --- |
| Clock sanity | doctor | doctor |
| Identity load + perms | doctor | doctor |
| UDP ephemeral bind | doctor | doctor |
| Ticket expiry scan | doctor | doctor |
| Discovery config echo | doctor | doctor |
| **Relay reachability** | doctor | `portl status --relay` |
| **DNS discovery probe** | (not done) | `portl status --relay` (attempts actual handshake) |

`portl doctor` is offline-safe, deterministic, and completes in
milliseconds. `portl status` is the command that actually talks to
the network.

## 12. Cleanup detail: `portl init` and `portl install`

### 12.1 `portl init`

```
portl init [--force] [--role operator|agent]
```

One command; composes:

1. If no identity, generate + persist (`id new`-equivalent).
2. Run `portl doctor`; pretty-print the results.
3. If `--role agent` or autodetected as a long-running host (systemd
   present, non-laptop hostname, `$PORTL_HOME` writable under
   `/var/`), suggest `portl install`.
4. Print the first-mint cookbook:
   `portl docker run <image>` / `portl slicer run <image>`.

### 12.2 `portl install`

```
portl install [TARGET] [--apply] [--yes] [--detect] [--dry-run]
```

Targets: `systemd`, `launchd`, `dockerfile`, `openrc`. When `TARGET`
is omitted, the command autodetects (see `§12.3`) and either prompts
to confirm or, with `--yes`, proceeds. `--detect` prints a JSON
description of what was found and exits 0.

With `--apply`, the command:

- Copies `current_exe()` to the target install path.
- Writes the unit / plist / script to the system location.
- Validates with `systemd-analyze verify` / `plutil -lint` / etc.
- Reloads the init system and starts the service.
- Tails the service log for 5s to confirm startup.

### 12.3 Autodetect heuristics

In order of strength, picking the first match:

| Signal | Target |
| --- | --- |
| `/bin/launchctl` exists + `uname -s` is `Darwin` | `launchd` |
| `/.dockerenv` exists or `/proc/1/cgroup` contains container markers | refuse, suggest `portl docker bake` or `--force` |
| `/run/systemd/system/` is a directory | `systemd` |
| `/sbin/openrc-run` exists | `openrc` (warn: less tested) |
| otherwise | fail with explicit list |

Running as non-root on a systemd host picks user-level systemd
(`~/.config/systemd/user/portl-agent.service`) instead of refusing.

## 13. Migration from v0.1

v0.2 is a breaking release. v0.1 operators need to do the following
once, on each host:

1. Back up `$PORTL_HOME/identity.bin`. Nothing else matters.
2. Stop any v0.1 agents (`systemctl stop portl-agent`, `docker stop`,
   etc.).
3. Install v0.2: `curl -L ...portl-v0.2.0-...tar.zst | ...`; see
   `README.md`.
4. For bare-host agents: `portl install systemd --apply`. The old
   unit is overwritten; `ExecStart` changes from
   `/usr/local/bin/portl agent run` to `/usr/local/bin/portl-agent`.
5. For the docker adapter: `portl docker run <image>` replaces
   `portl docker container add --image <image>`. No image rebuild
   required (orchestrate mode).
6. For the slicer adapter: same shape; `portl slicer run` replaces
   `portl slicer container add`.
7. Alias DB conversion: on first startup, if v0.2 finds
   `$PORTL_HOME/aliases.db` and no `aliases.json`, it reads the
   SQLite rows and writes the JSON file, then deletes the `.db`.
   One-shot, no rollback. v0.1 binary will not function against the
   v0.2 state directory after this point, which is acceptable —
   this is a major version bump.

Identity files are untouched across the transition; tickets minted
under v0.1 continue to verify because the ticket schema, canonical
form, ALPNs, and handshake are all unchanged.

Revocation JSONL is preserved byte-for-byte.

## 14. Risks and mitigations

| Risk | Mitigation |
| --- | --- |
| `docker cp` + `docker exec` injection breaks on some container runtimes (containerd shimv2, Podman rootless, Docker Desktop with userns-remap) | Exercise all three in CI on ubuntu-latest + podman-static; surface a clear error when the injection path fails; bake mode as escape hatch |
| SQLite → JSON alias store concurrency bugs on shared `$PORTL_HOME` | Serialized writes via `fs::rename` (atomic on every POSIX FS); document that two concurrent CLIs updating the same `$PORTL_HOME` is undefined (same contract as v0.1, which had SQLite's WAL for the same purpose) |
| Env-only config loses something an operator actually needed | Audit `agent.toml` usage across the known internal deployments before merging. If a field is in use and has no env mapping, add one rather than reintroducing TOML |
| Foreign-arch `--from-release` download on a sandboxed host with no internet | Detect immediately (DNS resolve + HEAD); fail with `--from-binary` suggestion |
| macOS release-mode crash is *not* caused by rusqlite and persists | Bisect between v0.1.0 and first-v0.2 commit once rusqlite removal lands; worst case, revert one flag and keep the JSON store anyway |
| Gateway users skip the migration and end up with a dead slicer deployment | Loud banner in v0.2 release notes; `portl-agent` in v0.2 rejects `--mode gateway` with a one-line "use `portl-gateway` now" error |

## 16. Acceptance criteria

A v0.2.0 build is releasable when all of the following hold:

1. `portl --help` lists the exact command set in `§4`.
2. `portl-agent --help` prints usage for a daemon with no
   subcommands.
3. `cargo tree --workspace -e normal` does not list `rusqlite`,
   `libsqlite3-sys`, `basic-toml`, or `wiremock` under `portl-agent`.
4. `portl init && portl docker run alpine:3 --caps shell` from a
   fresh host opens a shell end-to-end.
5. `portl install --apply` on vn3 (Ubuntu 24.04 with systemd) brings
   up a service that passes `portl doctor` immediately and answers
   `portl status` within 2 s.
6. `portl install --apply` on a macOS laptop installs a LaunchAgent
   that same behaviour holds for.
7. Release-mode `portl exec` on macOS arm64 completes without
   SIGABRT against a real ticket.
8. Migration from v0.1 described in `§13` is validated manually on
   one host per platform and the procedure is captured in
   `CHANGELOG.md`.
9. All acceptance tests added to the workspace pass under
   `cargo test --workspace --all-features` on both macOS arm64 and
   linux x86_64.
10. `docs/design/080-cli.md`, `090-config.md`, and the post-v0.1
    section of `120-roadmap.md` are either deleted or prefixed with
    a "Superseded by `140`" banner and point at this doc.
